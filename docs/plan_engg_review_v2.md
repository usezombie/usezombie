## Key Interactions to Verify

### Skill flow (Claude Code host)

1. User runs `/install-platform-ops` in a target repo.
2. Skill detects `.github/workflows/*.yml`, `fly.toml`, `Dockerfile` → infers deploy target.
3. Skill prompts for `slack_channel`, `prod_branch_glob`, `cron_opt_in` (deploy_target may be inferred or confirmed).
4. Skill resolves credentials in order: 1Password CLI (`op read`) → env vars → interactive prompt.
5. Skill writes `.usezombie/platform-ops/{SKILL,TRIGGER,README}.md` with substituted variables.
6. Skill calls `zombiectl install --from .usezombie/platform-ops/`.
7. Skill calls `zombiectl steer {id}` and opens an interactive session.
8. User types "morning health check" → zombie posts evidenced diagnosis to the configured Slack channel.

### Skill flow (multi-agent host — Codex CLI / OpenCode / Amp)

Same flow as above, but variable resolution happens through host-neutral natural-language Q&A (per A1 decision) rather than Claude's `AskUserQuestion`. Verify the same `.usezombie/platform-ops/` files are produced.

### Hosted-zombid default path (per A4)

When `zombiectl info` returns no reachable local zombied-api, the skill defaults to the hosted endpoint (`api.usezombie.com` or equivalent), prompts user to sign in via Clerk if not authenticated, and proceeds without requiring local infrastructure.

### Self-host opt-in path (per A3, A4)

User can explicitly choose self-host during the skill flow. Skill prints the `docs.usezombie.com/self-host` runbook URL and exits. User follows runbook on a clean Ubuntu 24.04 VM, brings up zombied-api, re-runs skill, install completes.

## Edge Cases

### Repo-detection edge cases

- Repo has no `.github/workflows/` → skill bails with "GitHub Actions detection required — non-GH CI is in a future version" (per OQ3).
- Repo has multiple `*.yml` workflows with different deploy targets → skill prompts user to pick one.
- Repo has `fly.toml` but no `.github/workflows/` → still bails (CI detection is the gate, not deploy target).
- Monorepo with multiple `package.json` files at different paths → skill picks the deepest one matching a workflow path.

### Credential resolution edge cases

- 1Password CLI installed but user not signed in → skill prints `op signin` instructions, retries.
- Env var present but empty → skill ignores and falls through to interactive.
- Interactive prompt receives empty input twice → skill aborts with clear message.
- Token shape mismatches expected format (e.g., user pastes Fly read-write token instead of read-only) → skill cannot validate but prints a warning that token scope matters per the README.

### File-generation edge cases

- `.usezombie/platform-ops/` already exists → skill refuses without `--force`.
- Target repo is a worktree → skill writes to the worktree path, not the main checkout.
- Generated SKILL.md contains a substitution that looks token-shaped → **gitleaks scan blocks the test** (per Q4).
- Network unavailable during template fetch → skill uses cache from prior run if available, else fails clearly.

### Install / chat edge cases

- `zombiectl install` fails with non-zero exit → skill prints stderr verbatim, exits without opening chat.
- User Ctrl-C's during chat → CLI exits cleanly, zombie keeps running (per M37_001 step 9).
- User re-runs `/install-platform-ops` after a successful install → skill detects existing `.usezombie/platform-ops/`, asks "update or skip?" (does NOT silently overwrite without `--force`).

### Template-fetch edge cases (per A5)

- Cache hit, but cache version older than skill's pinned tag → skill refetches.
- Cache hit, network down, version matches → use cache (offline mode works).
- Cache miss, network down → fail clearly with "no template cached and unable to fetch — try again when online."
- Pinned tag deleted on GitHub (rare) → fall back to `main`, warn user that the version is unverified.

## Critical Paths (must work end-to-end before launch)

1. **Cold install on Customer Zero's broken dev CD** — Day 1-3 dogfood. The launch demo screenshot.
2. **Cold install + first chat against hosted zombid** — the default OSS user experience. Should complete in <60 seconds from skill invocation to first Slack post.
3. **Cold install + first chat in self-host mode on clean Ubuntu 24.04** — the validated-self-host claim from A3. Block launch on this passing.
4. **Re-run idempotency** — running the skill twice on the same repo doesn't break anything. Refuses without `--force`.
5. **Secret-leak guardrail** — gitleaks scans every generated file in CI. Zero matches required.
6. **Multi-agent compatibility smoke** — same skill produces equivalent files in Claude Code, Codex CLI, OpenCode, Amp. Equivalence = same `.usezombie/platform-ops/SKILL.md` byte-for-byte after running through each host.
7. **Eval suite (in `usezombie/skills-evals`)** — given 10 fixture broken-CD logs, the platform-ops zombie's diagnosis names the right symptom ≥7/10 of the time.
8. **Context-lifecycle smoke** — a zombie that runs 25 tool calls in one stage gracefully uses memory_store at 5-call intervals (per A8 mechanism #2). Verify memory entries appear in `core.zombie_events` log.

## Regression Surface (must not break)

- M37_001 platform-ops sample (`samples/platform-ops/{SKILL,TRIGGER,README}.md`) is byte-identical before and after install — the skill consumes, never modifies.
- M19_003 `zombiectl install --from <path>` behavior unchanged.
- Existing CLI commands (`zombiectl credential add`, `zombiectl list`, `zombiectl status`, `zombiectl kill`, `zombiectl logs`, `zombiectl doctor`, `zombiectl install`) behaviors unchanged. NEW commands shipped in M33_001 substrate: `zombiectl steer {id}` (CLI did not exist before), and extended behavior on `zombiectl doctor` (now also checks workspace binding, per A9 resolution).
- `core.zombie_events` schema unchanged.
