## What changed

### Lead Zombie — v2 core architecture ships

- **`zombiectl install lead-collector && zombiectl up`** — two commands, running Zombie. A developer with no prior UseZombie knowledge can go from zero to a live agent receiving inbound email webhooks in under 2 minutes.
- **Zombie config format**: YAML frontmatter (machine config: trigger, skills, credentials, budget) + markdown body (agent instructions). Supports voice-transcribed instructions fed as the instruction body. CLI compiles YAML → JSON before upload; the server only ever sees JSON.
- **Webhook ingestion API**: `POST /v1/webhooks/{source}` — receives inbound events, Bearer-token auth (per-zombie), idempotency via Redis `SET NX EX 86400`, enqueues to `zombie:{id}:events` Redis stream. Returns 202 Accepted or 200 Duplicate.
- **ActivityStream**: append-only event log in `core.activity_events`. `logEvent` is fire-and-forget (never blocks the event loop). `queryByZombie` / `queryByWorkspace` with cursor-based pagination (`created_at DESC`).
- **Zombie event loop** (`src/zombie/event_loop.zig`): claims zombie from Redis stream, loads config and session checkpoint from Postgres, processes events via NullClaw agent in bwrap+landlock sandbox, checkpoints state after each event.
- **Credential injection**: agentmail API key and any `op://` credentials are resolved at runtime from vault and injected into the sandbox environment. No credentials ever in config files.
- **Session checkpoint**: `core.zombie_sessions` stores the last conversation context as JSONB, upserted after each event delivery. Resumable across restarts.
- **`zombiectl` CLI**: new Zombie-native subcommands — `install`, `up`, `status`, `kill`, `logs`, `credential add`, `credential list`. Old harness-era commands remain unchanged (cleaned up in M1_002).
- **Sample workspace**: `samples/lead-collector/` ships a pre-configured demo workspace with agentmail credentials stub so the hero demo runs with zero manual credential setup.
- **agentmail domain** added to executor network allowlist (`api.agentmail.to`).

### Schema additions (migrations 022–024)

- `core.zombies` (022): id (UUIDv7), workspace_id FK, name, source_markdown TEXT, config_json JSONB, status (active/paused/stopped), created_at/updated_at BIGINT. Unique per (workspace_id, name).
- `core.zombie_sessions` (023): one row per zombie, upserted after each event delivery. Stores context_json JSONB and checkpoint_at BIGINT.
- `core.activity_events` (024): append-only (UPDATE/DELETE blocked by trigger). Indexed on (zombie_id, created_at DESC) and (workspace_id, created_at DESC).

## Technical details

- Webhook dedup key format: `webhook:dedup:{source}:{event_id}` — 24-hour TTL.
- Zombie event stream key: `zombie:{zombie_id}:events` — `MAXLEN ~ 10000` (approximate trim).
- ZombieConfig parser validates skills against a known set (agentmail, slack, github, git, linear, cloudflare, pagerduty) plus pinned `clawhub://` refs (version number required, `latest` rejected).
- All credential values must start with `op://` — bare strings rejected at parse time.
- `extractZombieInstructions` returns a borrowed slice into `source_markdown` — no allocation. The markdown body after the YAML frontmatter `---` delimiter is the agent instruction text.
- Executor network allowlist: `agentmail.to` added alongside existing `github.com`, `api.anthropic.com`.
- `logEvent` swallows all errors — a Postgres failure on the write path does not surface to the event loop. Error is logged to stderr only.
- Cross-compile verified: `zig build -Dtarget=x86_64-linux` and `zig build -Dtarget=aarch64-linux`.

## Migration

- Migrations 022–024 are applied automatically by `zombied migrate` (or on startup if auto-migrate is enabled).
- No ALTER on existing tables — all new. Full teardown-rebuild is safe until v0.5.0 (schema freeze at v0.5.0).
- `zombiectl/package.json` version bumped from `0.3.1` → `0.5.0`. Run `npm install` or `bun install` in `zombiectl/` after pulling.

## Infrastructure notes

- Worker fleet: no changes required. The zombie event loop runs on the existing worker nodes.
- Redis: no new streams or keys beyond the webhook dedup key and zombie event stream. No schema changes to the existing run stream (`core:runs`).
- Postgres: three new tables, all in the `core` schema. GRANTs are in the migration files.
