# v0.7.0

**Date:** Apr 10, 2026
**Milestones:** M2_002, M5_001
**Specs:** M2_002_CLAWHUB_SKILL_FORMAT.md, M5_001_TOOLS_AS_ATTACHMENTS.md

## What changed

### ClaHub Skill Format (M2_002)

- Zombies are now directories (`SKILL.md` + `TRIGGER.md`) instead of single `.md` files
- `SKILL.md` follows ClaHub standard — publishable to ClaHub registry as-is
- `TRIGGER.md` carries deployment config (trigger, chain, budget, network, credentials)
- `zombiectl install` creates a directory with both files
- `zombiectl up` sends both files raw to the API — no client-side YAML parsing
- Deleted `simpleYamlParse`, `parseZombieMarkdown`, `parseYamlValue` (P1 greptile fix)
- Server now parses TRIGGER.md frontmatter into config_json (trusted, not client-supplied)
- Webhook auth now supports URL-embedded secret (`/v1/webhooks/{id}/{secret}`) with Bearer fallback
- Credentials are names (e.g. `agentmail_api_key`), not `op://` vault paths
- `chain:` field declares downstream zombies for event forwarding
- `skill:` field stores ClaHub registry references (resolution deferred to M3)
- Lint gate tightened from 500 to 400 lines per file

### Tools as Attachments (M5_001)

- Skills are dynamic: the NullClaw agent reads SKILL.md instructions and uses built-in tools (shell, HTTP, file_read) to interact with skill APIs. No compiled Zig per skill.
- Executor NullClaw built-in tool selection is now table-driven (`tool_bridge.zig`): adding a new built-in requires one registry entry, not a hardcoded if/else.
- Git credential setup extracted to `runner_credentials.zig` (400-line gate).
- Samples converted: `samples/lead-collector/` now uses `SKILL.md` + `TRIGGER.md` directory format (was single `.md` file).
- Three unreachable orphaned test files deleted (`proposals_test.zig`, `gate_loop_integration_test.zig`, `topology_gate_test.zig`).

## Architecture decisions

- **No vtable interface for skills.** Skills are config (TRIGGER.md declares domains, credentials, budget). The agent figures out how to use APIs from SKILL.md instructions + NullClaw's built-in shell/HTTP tools.
- **No credential_key on SkillEntry.** Credential resolution is the agent's job — it reads natural language instructions, searches the vault, and uses the approval gate when uncertain.
- **No `tool_registry.zig`.** Skill name validation and domain defaults come from `TRIGGER.md` `network.allow`. No compiled Zig per skill, ever.

## Technical details

- New Zig modules: `yaml_frontmatter.zig` (YAML→JSON converter), `config_helpers.zig` (trigger/budget/network parsers), `route_matchers.zig` (URL pattern matching)
- `YamlToJsonConverter` struct handles deferred bracket emission for correct array/object detection
- `WebhookRoute` struct carries `zombie_id` + optional `secret` from URL path
- `ZombieConfig` struct extended with `skill: ?[]const u8` and `chain: []const []const u8`
- `ZombieTriggerType` enum extended with `chain` variant
- Schema: `core.zombies` gains `trigger_markdown TEXT` and `webhook_secret_ref TEXT` (vault reference, not plaintext)
- Webhook secret stored in `vault.secrets` via `crypto_store`, resolved at auth time
- `src/executor/tool_bridge.zig` (new): table-driven NullClaw built-in tool registry. `resolve(name)` + `buildTools(alloc, spec, workspace, cfg)`.
- `src/executor/runner_credentials.zig` (new): extracted `prepareGitCredential()` from runner.zig.
- `src/executor/runner.zig`: reduced from 481 → 393 lines.

## Breaking changes

- `POST /v1/zombies/` body changed: `config_json` replaced by `trigger_markdown`
- `POST /v1/webhooks/{zombie_id}/{secret}` is the new preferred webhook URL format
- Credentials in TRIGGER.md must be names, not `op://` paths (validation rejects `op://`)
