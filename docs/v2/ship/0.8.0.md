## What changed

### Zombie directory format (M2_002)

- Zombies are now directories (`SKILL.md` + `TRIGGER.md`) instead of single `.md` files
- `SKILL.md` follows ClaHub standard — publishable to ClaHub registry as-is
- `TRIGGER.md` carries deployment config (trigger, chain, budget, network, credentials)
- `zombiectl install` creates a directory with both files; `zombiectl up` sends both files raw to the API
- Server now parses TRIGGER.md frontmatter into config_json (trusted, not client-supplied)
- Webhook auth now supports URL-embedded secret (`/v1/webhooks/{id}/{secret}`) with Bearer fallback
- Credentials are names (e.g. `agentmail_api_key`), not `op://` vault paths
- `chain:` field declares downstream zombies for event forwarding

### Dynamic skills & tool bridge (M5_001)

- Skills are dynamic: the NullCraw agent reads SKILL.md instructions and uses built-in tools (shell, HTTP, file_read) to interact with skill APIs — no compiled Zig per skill
- Executor NullCraw built-in tool selection is now table-driven (`tool_bridge.zig`): adding a new built-in requires one registry entry, not a hardcoded if/else
- Git credential setup extracted to `runner_credentials.zig`
- Samples converted: `samples/lead-collector/` now uses `SKILL.md` + `TRIGGER.md` directory format

### AI Firewall policy engine (M6_001)

- Every outbound request from a Zombie now passes through a 4-layer AI Firewall before reaching external APIs
- **Domain allowlist** — only domains declared in the Zombie's skills can be reached
- **Endpoint policy** — per-endpoint rules in TRIGGER.md `firewall:` section
- **Prompt injection detection** — scans outbound request bodies for instruction override, role hijacking, data exfiltration, and jailbreak patterns (with unicode normalization)
- **Content scanning** — inspects response bodies for credential leakage, PII (credit cards, SSNs), and API key patterns
- Every firewall decision (allow, block, flag) is logged as a structured event in the activity stream
- Fail-closed on all error paths

### Pipeline v1 removal (M10_001)

- All `/v1/runs/*` and `/v1/specs` endpoints now return HTTP 410 Gone with `ERR_PIPELINE_V1_REMOVED`
- `src/pipeline/` (v1 GitHub PR-solver, 28K lines) deleted — worker binary runs zombie threads only
- Pipeline tables replaced with `SELECT 1;` version markers (teardown-rebuild pattern)
- `SqlStatementSplitter` extracted with 14 unit tests — handles SQL comments with semicolons/apostrophes correctly

### API error standardization (M11_001)

- All error responses use `application/problem+json` format with `UZ-` prefixed error codes
- Error codes are table-driven via `error_table.zig` — HTTP status derived from code, not passed by caller
- `common.errorResponse(res, code, detail, req_id)` is the single error path (4-arg, no status param)

### Handler context wrapper (M11_002)

- New `Hx` struct eliminates 9-line boilerplate ritual from every authenticated handler
- `authenticated()` comptime wrapper: arena setup, request ID, auth check — all in one place
- `authenticatedWithParam()` variant for handlers with path params
- All ~42 `common.authenticate` call sites consolidated into the two wrappers
- Streaming and HMAC-verified handlers excluded (documented with comments)

## Technical details

- `src/zombie/firewall/` (new, 6 files, 946 lines): 4-layer firewall orchestrator
- `src/executor/tool_bridge.zig` (new): table-driven built-in tool registry
- `src/executor/runner_credentials.zig` (new): extracted git credential setup
- `src/http/handlers/hx.zig` (new): `Hx` struct + `authenticated()` + `authenticatedWithParam()`
- `src/errors/error_table.zig` (new): `UZ-` error code → HTTP status lookup via `StaticStringMap`
- Schema: `core.zombies` gains `trigger_markdown TEXT` and `webhook_secret_ref TEXT`
- Firewall events reuse `core.activity_events` with new event_type values

## Breaking changes

- `POST /v1/zombies/` body changed: `config_json` replaced by `trigger_markdown`
- `POST /v1/webhooks/{zombie_id}/{secret}` is the new preferred webhook URL format
- Credentials in TRIGGER.md must be names, not `op://` paths (validation rejects `op://`)
- All `/v1/runs/*` and `/v1/specs` endpoints removed (return 410 Gone)

## Migration

- None — tables rebuilt from scratch (pre-v1.0 teardown-rebuild policy)
