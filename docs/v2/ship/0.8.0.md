# v0.8.0

**Date:** Apr 11, 2026
**Milestone:** M6_001
**Spec:** M6_001_AI_FIREWALL_POLICY_ENGINE.md

## What changed

- Every outbound request from a Zombie now passes through a 4-layer AI Firewall before reaching external APIs
- **Domain allowlist** — only domains declared in the Zombie's skills can be reached (defense-in-depth on top of network policy)
- **Endpoint policy** — per-endpoint rules in TRIGGER.md `firewall:` section (e.g., "Stripe: allow GET /v1/charges, deny POST /v1/refunds without approval")
- **Prompt injection detection** — scans outbound request bodies for instruction override, role hijacking, data exfiltration, and jailbreak patterns (with unicode normalization)
- **Content scanning** — inspects response bodies for credential leakage, PII (credit cards, SSNs), and API key patterns from other services
- Every firewall decision (allow, block, flag) is logged as a structured event in the activity stream
- Fail-closed on all error paths — if the firewall errors, the request is denied

## Technical details

- New module: `src/zombie/firewall/` with 6 files (946 lines total):
  - `domain_policy.zig` (66L) — case-insensitive exact domain match, no suffix matching
  - `endpoint_policy.zig` (221L) — glob-based path matching, JSON rule parser
  - `injection_detector.zig` (175L) — 19 patterns across 4 categories, unicode `\uXXXX` normalization
  - `content_scanner.zig` (244L) — credit card, SSN, API key prefix detection, credential echo check, 1MB scan limit
  - `firewall.zig` (138L) — orchestrator combining all 4 layers with event type mapping
  - `firewall_test.zig` (110L) — orchestrator integration tests
- Uses `union(enum)` for `FirewallDecision`, `InjectionResult`, `ScanResult` — exhaustive switch enforcement
- Firewall events reuse existing `core.activity_events` table with new event_type values: `firewall_request_allowed`, `firewall_request_blocked`, `firewall_injection_detected`, `firewall_content_flagged`, `firewall_approval_triggered`
- Pattern matching is O(n*m) substring scan — target < 1ms for 10KB bodies

## Breaking changes

- None. Firewall is additive — existing tool bridge and activity stream interfaces unchanged.

## Migration

- None — no new tables. Firewall events use existing `core.activity_events` with new event_type values.
