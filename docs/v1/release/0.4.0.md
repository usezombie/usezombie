# v0.4.0

**Date:** Apr 05, 2026
**Milestones:** M21_001, M22_001

## What changed

### M21_001 — Agent Interrupt and Steer

- Users can now steer a running agent mid-run without aborting it via `POST /v1/runs/{id}:interrupt`
- Two interrupt modes: **queued** (delivered at next gate checkpoint) and **instant** (IPC delivery, falls back to queued in v1)
- New `ABORTED` terminal state for explicit abort flows (`/stop` from CLI, abort from UI)
- CLI command: `zombiectl runs interrupt <run_id> <message> [--mode=queued|instant]`
- SSE stream emits `interrupt_ack` event when the server receives an interrupt
- Watch mode (`--watch`) displays `-> steered (mode: queued)` inline in the transcript on interrupt receipt

### M22_001 — SSE Stream Correctness

- `zombied run --watch` now streams gate results in real time instead of buffering until run completion
- SSE heartbeat fires reliably every 20s (below the 25s socket timeout and 30s proxy idle cutoff)
- Reconnect with `Last-Event-ID` replays only missed events — no duplicate floods
- Run completing during the subscribe window no longer causes the stream to hang indefinitely
- `zombiectl run --spec <file> --watch` streams gate results with retry, Ctrl+C support, and idle progress indicator
- `zombiectl runs replay <run_id>` prints a per-gate narrative for completed runs

## Technical details

### M21_001
- Interrupt messages stored in Redis with `run:interrupt:{run_id}` key prefix, 300s TTL, last-write-wins
- Gate loop polls Redis via `GETDEL` (atomic read-and-delete) at the top of each iteration for exactly-once delivery
- `InjectUserMessage` executor IPC method wired for future instant-mode delivery (v2)
- State machine: ABORTED transitions allowed from all active states (SPEC_QUEUED through PR_PREPARED)
- 4 new Prometheus counters: `interrupt_queued_total`, `interrupt_instant_total`, `interrupt_fallback_total`, `run_aborted_total`
- `agent_id` added to `GateLoopConfig` and all interrupt log lines for full observability traceability
- Error codes: `UZ-RUN-008` (Redis store failure), `UZ-RUN-009` (run not in interruptible state)
- Input validation: 4096 byte message limit, mode enum validation, workspace auth gating

### M22_001
- §1: CLI uses `client.open()` + `req.reader()` incremental read loop instead of buffered `client.fetch()`
- §2: `SO_RCVTIMEO` (25s) on Redis pub/sub socket; heartbeat interval set to 20s so the first timeout wakeup always fires before the 30s proxy idle cutoff (invariant: heartbeat < SO_RCVTIMEO < proxy)
- §3: Unified SSE event ID namespace — `created_at` (Unix ms) used for both stored replay and live events; worker publishes `created_at` in gate event JSON
- §4: Post-subscribe race fix — re-query run state after `subscriber.subscribe()`, replay missed events if terminal
- §5: npm CLI `run_watch.js` module with `AbortController`-based SIGINT handling, 3 retries with `Last-Event-ID`, 30s idle progress indicator
- Bug fix: `sendCommand` in Redis pub/sub subscriber now flushes the socket writer after TLS encryption; without this, AUTH and SUBSCRIBE commands were never sent to Redis causing the stream handler to block indefinitely
- Bug fix: `zombiectl runs replay` reads `gate_results` field (matching server `ReplayResponse`); was reading `gates` which is always undefined
- CI: explicit `docker compose cp redis:/tls/server.crt` step ensures Redis TLS cert is available for integration tests; previously relying on Makefile `$(shell ...)` caused tests to silently skip

## Breaking changes

- SSE `id:` field on live events changed from sequential counter (`1, 2, 3...`) to `created_at` Unix milliseconds. Clients parsing `Last-Event-ID` as a sequence number must update to treat it as a timestamp.
- `streamViaPoll` fallback also uses `created_at` as event ID (was sequential counter).

## Migration

- None — tables rebuilt from scratch. No ALTER needed.
