## Problem Statement

Operational outcomes fall into limbo. When a deploy fails, an inference endpoint starts returning 404, or production looks unhealthy, the SRE manually correlates signals across 5+ tools (Grafana, SigNoz, Zabbix or equivalent, app logs via gRPC observability with tokens in 1Password, GitHub Actions runs, SSH-to-baremetal), holds the timeline in their head, frequently restarts services with no understood root cause, and has no automatic mechanism to keep customers or stakeholders informed during the incident.

For the v2 MVP, the sharpest, datedmost-recent instance of this pain is **deploy and general production-outage triage**. Customer Zero (the author) has a currently-broken usezombie dev CD pipeline ([GH Actions run #24861966717](https://github.com/usezombie/usezombie/actions/runs/24861966717/job/72790030492#logs)) and an SRE workflow at E2E Networks where outage triage is the cursed step. The same pattern recurred at gettune.com (2020s) — AWS Event Queue down, S3 PDF upload failures, customer-facing comms gaps, signal-correlation witch hunts.

The wedge is not "fix all SRE problems." It is: **let any small-team operator install a markdown-defined, durable, BYOK, self-hostable zombie that watches their GitHub Actions CD pipeline, reasons over failures, posts an evidenced diagnosis to Slack, and (later) extends into the bastion role — owning incident communication and customer-facing status.**

## Demand Evidence

Honest demand bucket (per D1): **just my own pain — no external test yet (option D)**. Caveats and adjacent signal:

- Customer Zero has dated, concrete pain right now (broken dev CD migration, real GH Actions URL).
- Past-life recurrence: at gettune.com (CTO, 2020s), the same outage-comms gap caused CEO escalations during AWS S3 / Event Queue outages. Pattern is not novel to E2E.
- Adjacent vendor TAM is real: AWS DevOps Agent went GA Mar 31, 2026; Sonarly (YC W26) shipped; incident.io ships AI SRE; Atlassian Statuspage ships AI auto-drafting (Feb 2026). The category is intensely validated by competitors — the question is differentiation, not whether anyone wants this.
- "Will pay" / "got angry when prototype broke" / "named human" demand: **none yet**. The honest assignment is to convert this to ≥3 named operators within 2 weeks of wedge ship.

## Status Quo (Customer Zero workflow today)

Recent work-outage example: user-facing agent reports DB calls slow → SSH to host → open Grafana → open observability gRPC endpoint (token fetched from 1Password) → switch to browser for Zabbix metrics in time window → check Django API logs → still no clear cause → **restart service with no clue why** (the SRE failure mode). 5+ tools, no shared timeline, no persistent memory across incidents, no automatic stakeholder communication.

The ONE worst step (per D3) is **cross-correlating signals across tools** — bouncing tabs while holding the timeline in one's head at 2am. Adjacent pain explicitly named: **outage communication is manual; customers complain they aren't informed**.

## Target User & Narrowest Wedge

**Persona** (per D4 + D7 sharpening):

- Primary: SRE / platform engineer / 10x indie engineer / small high-agency team running their own infra (OVH bare metal today; AWS VMs / DO droplets / on-prem GPU clusters at the broader category) who **cannot or will not pipe their full ops stack into another SaaS** (cost, security, data sovereignty, SaaS-fatigue).
- Secondary: agentic-OS / agent-economy operators where human-in-the-loop is minimal — small teams that need a durable runtime, not another vendor dashboard.
- Customer Zero: the author at E2E Networks, with a broken dev CD as the immediate dogfood surface.

**Narrowest wedge** (per D5, D8): **Approach B — a Claude Code skill `/install-platform-ops`** distributed via the `usezombie/usezombie` repo and droppable into `~/.claude/skills/` (also compatible with Amp, Codex CLI, OpenCode — it's a SKILL.md, not a bash script). The skill auto-detects the user's repo (CI provider, deploy target, Slack channel inferred from `.github/workflows/*.yml`, `fly.toml`, `Dockerfile`, etc.), asks 3 tight gating questions, generates a tailored `.usezombie/platform-ops/{SKILL,TRIGGER,README}.md` in the user's repo, helps resolve credentials, and drives `zombiectl install --from .usezombie/platform-ops/` + `zombiectl steer`.

The skill is the install UX. No bash one-liner (Approach A dropped — cross-platform shell-test cost > value when SKILL.md is already portable across multiple agent CLIs).

## Constraints

- Cannot out-spend Sonarly's YC funding or AWS's marketing budget. Differentiation must be structural (OSS / self-host / BYOK / markdown-defined), and distribution must be build-in-public + Customer Zero + open-source GitHub stars.
- M37_001 platform-ops sample (SKILL.md + TRIGGER.md + README.md) already shipped — the skill must consume that, not replace it.
- M19_003 (`zombiectl install --from <path>`) shipped — the skill leans on it.
- "Self-hosted" today is a partially-validated claim: 2 substitutions tested (Postgres, Redis), 3 unknowns (Clerk → env-token/OIDC, process orchestration on a vanilla Linux VM, KMS adapter), 1 risk-load-bearing unknown (executor's Landlock+cgroups+bwrap sandbox on non-Fly Linux). The runbook must label tested rows honestly.
- The skill must work without exhausting users with questions — strict ≤4 AskUserQuestion gates total.

## Premises

Accepted in this session (per D7), with the noted sharpenings:

- **P1**: TAM is "operators at teams that cannot/will not pipe their ops stack into another SaaS," explicitly including AI-infra companies, GPU cloud providers, regulated mid-market, agentic-OS/agent-economy operators, 10x indie engineers, and small high-agency teams. NOT "all SREs everywhere."
- **P2**: Differentiation = OSS + self-hostable + BYOK + markdown-defined durable agent runtime. OSS is true today. Self-hostable requires a documented runbook — Postgres and Redis substitutions tested, Clerk-shim and vanilla-VM process orchestration NOT tested. Treat self-host as a doc + small-code workstream, not vapor.
- **P3**: Wedge C (GH-Actions-CD-failure responder) is the right first artifact, marketed as "*your* zombie running on *your* infra against *your* CI" — not "another GH-Agentic-Workflows-style add-on."
- **P4**: The bastion → automatic-statuspage extension is real and valuable but competes head-on with Atlassian Statuspage AI (Feb 2026) + Dust + PageCalm. Earn trust on internal-only incidents first; don't lead with customer-facing comms in v1.
- **P5**: Distribution = build-in-public + Customer-Zero dogfood + OSS GitHub stars. The skill itself is a public, screenshot-able artifact. Agents currently spin in OVH baremetal — that's the demo backdrop.

## Approaches Considered

### Approach A: Curl-pipe-bash installer (rejected)
Single shell script piped from GitHub raw URL. Wraps `zombiectl install --from samples/platform-ops` with credential prompts. **Rejected** because cross-platform shell testing (macOS / Linux flavors / WSL) eats founder time, and SKILL.md is already portable across Claude Code, Amp, Codex CLI, and OpenCode without a bash port.

### Approach B: Claude Code skill `/install-platform-ops` (chosen)
Gstack-style SKILL.md that detects the user's repo, asks 3 tight gating questions, generates tailored `.usezombie/platform-ops/{SKILL,TRIGGER,README}.md`, resolves credentials via `op` / env / interactive, calls `zombiectl install --from .usezombie/platform-ops/`, opens `zombiectl steer`. Effort M (~1 week human / ~2 hours CC). Reuses M19_003 + shipped `samples/platform-ops/`.

### Approach C: GitHub Action template + hosted demo (deferred)
`samples/.github/workflows/zombie-on-failure.yml` + `demo.usezombie.com`. Pull-based fit to the wedge, but pushes the hosted-vs-self-host decision before the wedge is validated. **Deferred to v2.5** as the second wedge once B has 5 external installs.

## Recommended Approach (revised post-Codex, 5-6 week milestone)

The original design proposed a 1-week packaging milestone. The eng review + Codex outside-voice surfaced that the substrate underneath is incomplete — the install contract is broken, M33 (worker control + chat) is pending, structured creds + network policy aren't wired through. Packaging on top of that substrate produces a Potemkin demo. Revised plan:

### Week 1-2: Substrate (launch-blocking — promote pending specs to active)

1. **Install API contract alignment.** Fix the CLI POST shape `{source_markdown, trigger_markdown}` ↔ API `{name, config_json, source_markdown}` mismatch. Reconcile `tools:` (sample) vs `skills:` (parser). One unit test per shape. (zombiectl/src/commands/zombie.js:83 ↔ src/http/handlers/zombies/api.zig:32 ↔ src/zombie/config_parser.zig:108)
2. **Auth fix + extend `zombiectl doctor`.** Remove `"zombie.install"` from `AUTH_EXEMPT_ROUTES` (zombiectl/src/cli.js:36). **Extend the existing `zombiectl doctor` command** (already auth-exempt, already a diagnostic surface — `routes.js:6`) to verify token validity + server reachability + active workspace + workspace binding. NO new `preflight` sibling command. The install-skill calls `zombiectl doctor --json` before invoking install; on failure it surfaces the doctor output verbatim. Applies to BOTH hosted users (Clerk OAuth token) AND self-host users (local API token). Plus optional cleanup chore: namespace `login`/`logout` under `zombiectl auth login` / `zombiectl auth logout` if not already done.
3. **Promote M33_001 (worker control stream + steer CLI) from `docs/v2/pending/` to `active/`.** Operator-initiated interaction surface. New zombies discoverable by the worker without restart. **CLI command is `zombiectl steer {id} [<message>]`** (NOT `chat` — that name was rejected; steer matches the existing server endpoint `POST /v1/.../zombies/{id}/steer` per `src/http/handlers/zombies/steer_integration_test.zig`). Interactive mode reads stdin, batch mode takes the message inline. M33_001's spec needs amending to use `steer` consistently. This IS the recommended interaction model — decided in office-hours D3 (the bastion framing).
4. **Promote M34_001 (event history + webhook ingestion) from pending to active.** Required for the GH Actions wedge: GH posts on `workflow_run.conclusion=failure` → arrives as synthetic event on `zombie:{id}:events` with `actor=webhook:github`. Same reasoning loop as a chat steer; just a different actor.
5. **Promote M35_001 (per-session policy + credential templating) from pending to active.** Wire structured `{host, api_token}` creds through `createExecution`. Make the executor receive tool config + network policy. Replace `http_request`'s empty `allowed_domains = &.{}`.
6. **Extend M37_001 platform-ops sample with a GH Actions webhook trigger path.** Today's TRIGGER.md says `trigger.type: chat` — extend to `trigger.types: [chat, webhook:github]` (or equivalent dual-trigger schema). Add `webhook.payload_schema` for the GH Actions failure event. SKILL.md prose teaches the agent to reason over GH run logs same way it reasons over fly logs — `http_request` GETs the failed run's logs via GitHub's API. **This is the wedge moment.**
7. **Optional: partial M36_001 (live watch + docs)** if needed for the launch demo's audit trail.

### Week 3-4: End-to-end validation

6. **Self-host validation on clean Ubuntu 24.04** (per A3) — install zombied-api on a fresh DO droplet or AWS VM, follow the `/self-host` runbook, run platform-ops end-to-end, document any breakage. Block launch on this passing.

- **P0** — Sample claims structured creds + per-host network policy + read-only tool boundaries; reality is single `--value` string credentials, no tool/network policy passed to executor, `http_request` built with empty `allowed_domains`. *Accepted — M35_001 (per-session policy + credential templating) promoted as Week 1-2 substrate work.*

- **P1** — Context-lifecycle plan uses fields not implemented in code (current checkpoint is just `last_event_id` + truncated `last_response`). *Accepted — only mechanism #1 (rolling tool-result window) ships in v1, since v1's chat-only flow is bounded ~5-10 tool calls. #2 + #3 ride along in M{N+1}.*
