## Why this milestone existed

Two unrelated wounds were still open after M11_005 shipped tenant-scoped billing.

The first was political more than technical: the `ZOMBIED_ADMIN_API_KEY`
/ `API_KEY` env-var path minted a principal with `role=admin,
tenant_id=null` so the founder could curl the API before Clerk signup
was wired. Post-M11_005 the only way an authenticated request could
reach `create_workspace` or the GitHub OAuth callback with a null
tenant_id was through that escape hatch — and those handlers papered
over the null by fabricating a tenant on the fly and calling
`provisionFreeDefault`. M11_005 landed both stopgaps with
`// TODO(legacy-bootstrap)` comments and we agreed to remove them the
moment the env-var went away. The user's framing was the cleanest
summary: *a politically stupid design*.

The second wound was silent. When a tenant's balance hit zero the
worker `info`-logged `metering.exhausted`, XACKed the event, and let
the zombie keep running on the operator's dime. No operator-visible
signal, no dashboard banner, no API flag, no halt. Pre-alpha this was
"fine"; alpha needs either an explicit "free runs continue" policy or
a concrete gate with observability behind it.

## Decisions at CHORE(open)

The spec originally hedged on admin auth: "Clerk JWT `role=admin` …
(or a new tenant-issued admin-flagged API key — decided during PLAN)."
Resolving that hedge in conversation with the user produced the
first set of non-obvious calls recorded here:

- **No "platform tenant" invariant.** My first draft proposed seeding
  a singleton `core.tenants` row as the home for the admin user so
  the "every authenticated request has non-null tenant_id" invariant
  would hold absolutely. The user pushed back — correctly — that
  this introduced new terminology ("platform tenant") for no payoff.
  The admin is just a regular tenant whose Clerk user has
  `publicMetadata.role=admin`. Everyone signs up the same way; the
  *only* difference for admin is a one-line manual Clerk Dashboard
  edit after signup.
- **No server-side admin allowlist.** My second draft proposed
  cross-checking `role=admin` claims against a whitelist of
  tenant_ids to prevent privilege escalation via publicMetadata.
  The user — again correctly — pointed out that this is theater: if
  an attacker can write publicMetadata.role, they can also write
  publicMetadata.tenant_id. The trust boundary is Clerk Dashboard
  access itself (1Password + 2FA); a second check adds no defense.
  Folded into Out of Scope.
- **Vault path shape.** The minted `zmb_t_…` admin key lives at
  `op://ZMB_CD_<env>/usezombie-admin` field `api_key` (single item,
  three fields: `username`, `credential`, `api_key`). Not a separate
  vault item. Captured in the playbook.

These are the kinds of design-level non-obviousnesses that future
maintainers will benefit from seeing reasoned through.

## The shape of the actual edits

§2 (schema + facade) and §4 (response surface) were mechanical —
add a nullable column in place (pre-v2.0 teardown, not ALTER),
thread `exhausted_at_ms: ?i64` through `BillingRow → Billing`, add
`markExhausted` as an atomic `UPDATE … WHERE
balance_exhausted_at IS NULL RETURNING …` (one SQL round-trip, no
read-then-write race), and surface `is_exhausted` + `exhausted_at`
on `GET /v1/tenants/me/billing` + the OpenAPI schema.

§3 (policy + worker gate) had the most ambient design choices.
`BALANCE_EXHAUSTED_POLICY={continue|warn|stop}` is parsed in
`src/config/balance_policy.zig` — its own module so the parser is
unit-testable without dragging in a whole config graph. The
pre-claim gate lives in `metering.shouldBlockDelivery` and only
fires under `stop`: it acquires a connection, resolves `tenant_id`
from `workspace_id`, reads `exhausted_at_ms`, and returns true
iff set. DB failures fail-open — the gate is a *feature*, not an
availability primitive; a Postgres blip must not become a delivery
outage. Activity events are written on the same conn the metering
path already holds (RULE CNX): `balance_exhausted_first_debit` on
the NULL→now transition (one-shot, gated by `markExhausted`'s
return value), `balance_exhausted` under `warn` (rate-limited
1/workspace/day via a same-conn `SELECT 1 FROM core.activity_events
… LIMIT 1` probe), `balance_gate_blocked` under `stop` per blocked
delivery. Rate-limiting through a query rather than in-memory state
is deliberately stateless-correct so restarting the worker doesn't
reset the counter.

Extracting the inline tests from `activity_stream.zig` into
`activity_stream_test.zig` was a RULE FLL forced move — my
additions pushed the file to 389 lines and I needed it under 350.
Splitting the tests out put it at 309 with room to grow.

§1 (bootstrap removal) was the biggest surface. The intent is
simple — delete everything — but the blast radius covered four
unrelated layers: middleware (delete `admin_api_key.zig`, env-var
branch in `bearer_or_api_key.zig`, registry field + chain +
accessor), config (delete `loadApiKeys`, `hasUsableApiKey`,
`MissingApiKey`, `InvalidApiKeyList`; add `OidcRequired` and make
OIDC mandatory), HTTP context (delete `Context.api_keys` and the
dead `common.authenticate` / `authenticateApiKey` / `matchApiKey`
chain that turned out to have no callers post-M18_002), test
harness (two copies of the registry-construction boilerplate —
`test_harness.zig` and `cross_workspace_idor_test.zig` — both
needed to drop `.api_keys`), doctor (stop probing an env-var that
no longer exists), `docker-compose.yml` (scrub `API_KEY:
local-dev-api-key`).

The scope decision I *didn't* make was removing the
`orelse generateTenantId(...)` fabrication paths in
`workspaces/lifecycle.zig` and `auth/github_callback.zig`. The
spec's original intent was to remove them with the env-var path
because the env-var was the only thing producing `tenant_id=null`
principals — but that's not quite true post-Clerk. The
signup-race window is real: after `user.created` fires we create
the `core.tenants` row in-process but the user's session JWT
doesn't yet carry `publicMetadata.tenant_id` because we don't
write it back to Clerk via the Backend API. Closing that loop
requires a new outbound HTTP client, `CLERK_SECRET_KEY` vault
wiring, retry + idempotency, and error-mapping — substantial
enough to warrant its own workstream. So: dims 1.2, 1.3, 1.7
punt to M11_006_b; the fabrication paths stay for now; the
handler comments are updated so the next reader knows *why* the
fallback exists and what would let them delete it.

## Playbook 012

The admin-bootstrap playbook is authored in this workstream but
runs manually per environment. Steps 1 (signup via website) and 2
(Clerk Dashboard role flip) are human-only and interactive; steps
3–6 (mint `zmb_t_…`, vault write, verify) are agent-executable
via `curl` + `op`. Rollback procedure is documented. The playbook
is not in the CI path — it runs when the operator is ready for
live testing.

- **`make memleak`.** Server lifecycle didn't change — only a
  middleware struct got smaller. The hot path (`bearer_or_api_key`)
  saw lines removed, not added. I'm citing CI for this rather than
  pasting a local run.
- **Greptile learnings capture.** No net-new rule surfaced — this
  was a deletion pass, not new territory.
