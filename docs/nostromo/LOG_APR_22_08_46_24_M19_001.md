## What I found before writing a line of code

Three load-bearing discoveries, each of which would have caused silent drift if I had just followed the spec's dimensions in order.

**1. M19's assumed API surface is mostly fictional.** The spec's §5.1 says *"All existing (zombiectl calls these today). UI consumes the same endpoints. This is a pure frontend milestone."* I verified against `src/http/route_manifest.zig` and against `zombiectl/src/commands/zombie.js`. What actually exists today:

- `POST|GET|DELETE /v1/workspaces/{ws}/zombies[/{id}]`
- `POST .../{id}/stop`
- `GET .../{id}/activity` / `/telemetry`
- `POST /v1/webhooks/{zombie_id}` (the public ingress — webhook URL is deterministic, no lookup endpoint needed)
- `GET /v1/tenants/me/billing`

Missing entirely: `GET /v1/skills`, `PATCH /zombies/{id}`, `GET|PUT /firewall`, `POST /schedule`, `:pause`, `:resume`, `GET /triggers`. `zombiectl` itself only implements install/list/delete/activity — the CLI subcommands the spec claims as equivalents (`zombie schedule`, `zombie firewall set`, `zombie triggers list`, `zombie update`) do not exist.

So the spec's scope was not deliverable as written. User's instruction was "no new spec" — so I amended M19_001 in place, trimmed to the buildable surface, and named the follow-up workstreams inline in an §0.5 amendment block.

**2. M11_006 has an unlabeled handoff to M19.** Its Out of Scope line says *"UI banners / dashboard affordances for exhausted state — separate M19 workstream pulls `is_exhausted` into the UI."* M19_001 as originally written has zero mention of `is_exhausted`. That's the classic drift — one spec marks a thing out-of-scope "for M19" while the M19 spec doesn't know it exists. I absorbed the handoff into this workstream (new §5) rather than leaving it as another sibling so it doesn't float unassigned.

**3. Route ownership collision with M27.** M19 owns panel components under `zombies/[id]/components/`; M27 owns `zombies/[id]/page.tsx` (the host file that imports them). M27 is blocked on M26_001 (design-system unification). Without the host file, M19's panels have nowhere to render and the install redirect lands on a 404. I scoped a minimal stub `page.tsx` as part of M19, tagged with a comment telling the dashboard-pages owner to replace it. Not beautiful, but the alternative was shipping code no one can see.

I surfaced all three findings to the user before touching any file. User approved the amendment + scope reduction + stub approach in one message, so I proceeded.
