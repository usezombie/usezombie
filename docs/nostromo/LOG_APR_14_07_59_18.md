### What was done

### Key decisions

**RULE FLL split (route_table_invoke.zig):**  
`route_table.zig` reached 452 lines after populating all 42 invoke functions inline. The `make lint` FLL gate caught it. Natural split: moved all `invokeXxx` functions to a new `route_table_invoke.zig` (290 lines). `route_table.zig` dropped to 170 lines. The invoke functions are `pub fn` accessed via `const invoke = @import("route_table_invoke.zig")` in the switch. Clean separation: types + specFor + tests in one file, method dispatch + handler calls in the other.

**Non-optional `principal` in Hx:**  
The spec originally implied `principal` could be optional. Making it optional would have forced `orelse` or `?` unwraps in 8+ handler files — unnecessarily invasive for Batch D. Kept `principal: common.AuthPrincipal` non-optional; routes with `none` policy receive a zero-value `.{ .mode = .api_key }`. Handlers on none-policy routes never access `hx.principal`, so this is safe. Documented in the Hx struct doc comment.

**`hx.*` copy semantics:**  
Inner handlers take `Hx` by value (e.g. `zombie_api.innerCreateZombie(hx.*, req)`). The `hx.*` dereference copies the Hx struct (5 fields, all pointer-sized) — cheap. This avoids changing every handler signature to `*Hx`. The response pointer `hx.res` is a pointer inside the struct, so writing to it from the copy still mutates the actual response. This is the existing contract — Batch D doesn't change it.

**`none` policy for several routes:**  
- `receive_webhook` — `webhook_url_secret` LookupFn is still a stub returning null. Using `webhookHmac()` policy would reject every webhook. Handler does its own secret check. Noted in route_table comment for Batch E follow-up.
- `github_callback`, `slack_callback` — OAuth state+nonce consumed by handler. Using `oauth_state` policy would double-consume the Redis nonce.
- `grant_approval_webhook` — Redis nonce consumed by handler; no standard policy applies.
- `execute` — zombie api_key validated by handler against DB; no bearer token.

**agent_relay.zig and slack_interactions.zig not touched:**  
Both are at/over 350 lines. Invoke functions for their routes live in `route_table_invoke.zig`. Handler files expose `handleXxx` (public functions) rather than `innerXxx` — Batch D's convention was to rename wrappers to `innerXxx` for handler files that previously used `hx.authenticated*`. Since `agent_relay` and `slack_interactions` never used the wrappers (they had their own auth patterns), their public API stayed as `handleXxx`. The invoke functions in `route_table_invoke.zig` call these directly.
