Single external caller (`src/cmd/serve.zig`) needed zero edits because the public surface (`ServeConfig`, `ValidationError`, `ServeConfig.printValidationError`) is preserved verbatim via re-exports.

### What actually happened

1. **Decomposed `ServeConfig.load()` (95L → 30L orchestrator + 5 sub-loaders).** Sub-loaders return intermediate "slice" structs (`SizesConfig`, `OidcConfig`, `EncryptionConfig`, `MiscConfig`) that the orchestrator assembles. Each sub-loader uses its own internal `errdefer` chain; the orchestrator threads one errdefer per produced section, so a late failure (e.g. encryption rejected) frees every prior section. This is materially cleaner than the original 95-line block where 6 separate `errdefer if (foo) |v| alloc.free(v)` lines accumulated and reading the cleanup contract required threading them mentally.

2. **One serve.zig signature gotcha caught by the build.** First façade draft promoted `printValidationError` to a top-level fn on the runtime module (`runtime.printValidationError(err)`). serve.zig calls `runtime_config.ServeConfig.printValidationError(err)` — i.e., it's a static member of `ServeConfig`. Caller surface preservation required nesting the alias inside the struct: `pub const printValidationError = validate.printValidationError;` inside ServeConfig. Trivial fix, zero serve.zig churn.

3. **Pub-audit caught dead code.** `runtime_loader.zig` initially had `freeMisc` for symmetry with `freeOidc` / `freeEncryption`. But `loadMisc` runs last in the orchestrator — there's no failure path between it and the assembled return, so no errdefer slot consumes `freeMisc`. Dead code by construction, removed per ZIG_RULES progressive cleanup.

4. **Façade-fan-out pattern reused** from M28_004 (now codified in `ZIG_RULES.md`). `runtime.zig` ends with a `test {}` block importing the three new `*_test.zig` files. `main.zig` had no entry for `config/runtime.zig` originally (it was reached transitively through `cmd/serve.zig`) — so test discovery now happens via the façade's `test {}` block, which gets compiled because `main.zig` already pulls in `cmd/serve.zig` → `config/runtime.zig`. Verified by checking the test count: all 14 ServeConfig.load tests + 5 env-parse + 2 validate ran.
