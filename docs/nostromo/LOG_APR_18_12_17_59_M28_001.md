# Ripley's Log ‚Äî M28_001 Unified Webhook Auth Middleware

## Scope recap

Consolidated webhook authentication ‚Äî previously scattered across `webhooks.zig` (URL secret + Bearer), `webhook_hmac.zig` (approval HMAC), and `slack_events.zig` (per-request env read + inline `verifySlackSignature`) ‚Äî into a single middleware that runs before handlers. Handler code now contains zero auth logic; new HMAC-SHA256 providers ship without touching Zig ‚Äî users declare the signature scheme in TRIGGER.md and the config parser wires it through.

## Decisions made

### 1. Filename deviation from spec

Spec called the middleware `webhook_auth.zig`. Renamed to `webhook_sig.zig` during EXECUTE because every code path in it is signature- or secret-matching driven ‚Äî "auth" was too broad and collided with the adjacent `auth/middleware/` directory's own semantic load (`clerk_bearer.zig`, `api_key_bearer.zig`, etc.). Signatures are the actual primitive.

Why this is worth recording: the spec is now the older artifact. Anyone searching for `webhook_auth.zig` based on the spec name will miss the file. Updated the spec's Files Changed table + Interfaces section to reflect the rename before moving to `done/`.

### 2. Comptime generic over *anyopaque (RULE NTE)

First cut of `WebhookSig` took `vault_lookup_ctx: *anyopaque` + a C-style function pointer. Worked, compiled, passed tests. Then caught myself: the host only ever passes `*pg.Pool`, and the `@ptrCast` on the callback side was silently type-unchecked ‚Äî a typo in the cast target would be a runtime memory-corruption bug that compiles clean.

Refactored to `pub fn WebhookSig(comptime LookupCtx: type) type`. Host passes `*pg.Pool` directly, compiler verifies the lookup fn's signature at the call site, zero runtime cost. This is the pattern `chain.Middleware(Ctx)` itself uses ‚Äî stayed consistent.

Codified as **RULE NTE** in `docs/greptile-learnings/RULES.md` (committed to dotfiles earlier this session ‚Äî commit `fcbd2fc` on `indykish/dotfiles`). The only legitimate `*anyopaque` in the chain layer is `chain.Middleware.ptr` itself (the chain runner needs a homogeneous array of mixed middleware types, which genuinely requires erasure). Everywhere else ‚Äî prefer comptime generics.

### 3. Extracted serve_webhook_lookup.zig for RULE FLL

The real DB+vault `WebhookLookupFn` implementation wanted to live in `serve.zig` next to where the middleware is constructed. That would have pushed `serve.zig` past the 350-line gate. Extracted to `src/cmd/serve_webhook_lookup.zig` (53 lines) ‚Äî acquires two pool connections (one for the `zombie` row lookup, one for vault decrypt), returns an optional `WebhookSecret`.

Gotcha for future pool-pressure analysis: two connections per webhook request is not new ‚Äî it mirrors the old inline handler code ‚Äî but it's now colocated in one file where it's easier to see. Worth remembering if we ever instrument webhook latency and find pool contention.

### 4. ZombieSession size assertion bump

Touched `src/zombie/event_loop_types.zig` ‚Äî `ZombieSession` size went from 320 ‚Üí 376 bytes because the new signature config fields are carried through on the session. The static size assertion had to be bumped. This will conflict with any parallel branch that also touches `event_loop_types.zig`, but as of this writing the only sibling M28 branch (`feat/m28-api-keys`, being renumbered to M28_002 per another agent) doesn't touch this file.

### 5. Spec vs rules conflict ‚Äî handled at PLAN

The spec's Interfaces block showed `vault_lookup_ctx: *anyopaque`. Under AGENTS.md "Spec-vs-rules conflict check," this conflicts with the broader "no type erasure when generics suffice" principle (later codified as RULE NTE). Amended the spec during EXECUTE to reflect the comptime-generic signature, in line with the rule: "spec is an instance, rules are the constant."

### 6. Slack handler dual verify removal

Found two copies of `verifySlackSignature` ‚Äî one in `slack_events.zig`, one in `slack_interactions.zig` (events webhook vs interactive components webhook). Both did the same inline env-read + HMAC check. Both now route through the middleware. The spec only mentioned the events copy; the interactions copy was caught by the orphan sweep during VERIFY. Updated the Dead Code Sweep table in the spec to reflect the dual removal.

## Dead ends / things tried and rejected

- **Keeping a thin `verifyWebhookAuth` wrapper in `webhooks.zig`** ‚Äî tried this first to minimize diff surface. Rejected because the goal of the milestone is "zero auth logic in handler"; a wrapper is still auth logic. Ripped it out, handler is now a pure dispatch-and-respond path.
- **Making the HMAC provider registry runtime-loaded from config** ‚Äî tempting for maximum flexibility, but it defeats the `@setEvalBranchQuota` + comptime invariant checks (unique `sig_header`, non-empty `sig_header`). Kept the registry `comptime []const VerifyConfig` so the invariants are enforced at build time, not runtime. Custom per-tenant providers still work via the `signature.header`/`signature.prefix` block in TRIGGER.md ‚Äî they just don't enter the registry proper.
- **Per-request env read for Slack signing secret** ‚Äî was explicitly one of the problems this milestone fixes. Slack secret now loaded at boot via `WebhookSignatureConfig` parsed from the main config.

## Trade-offs considered

- **Priority order of auth strategies** ‚Äî URL secret first, then HMAC, then Bearer. Reasoning: URL secret is explicit ("this request carries a secret"); HMAC is provider-fingerprinted (headers identify the provider); Bearer is the generic fallback. Reversing this (Bearer first) would make URL secret routes vulnerable to any valid Bearer token the tenant has ‚Äî wrong security posture.
- **Provider header collision (both `x-slack-signature` and `x-hub-signature-256` present)** ‚Äî first match in registry wins. Acceptable because this is a pathological case (no real sender produces both). Documented in Failure Modes.
- **Constant-time comparison leaks length but not value** ‚Äî per RULE CTM; acceptable for fixed-length webhook secrets. Noted in Platform constraints.
