## Decisions made

### 1. `:steer` auth breadth → broad, with audit-trail comment

The question: should `innerZombieSteer` additionally require `hx.principal.workspace_scope_id != null` (narrow, preserving M23_001 semantics) or trust `authorizeWorkspace` alone (broad, current)?

Picked broad. Reasoning:

- `authorizeWorkspace` is the canonical workspace-authZ gate in the post-M18_002 world. Every workspace-scoped handler in the codebase (zombie create/list/delete, activity, credentials, grants list/revoke) relies on it as the sole workspace-edge check.
- Adding a second `workspace_scope_id != null` gate on `:steer` would special-case one endpoint for no structural reason. If membership-based principals are trusted to list/delete zombies, they should be trusted to steer them.
- If operator-role gating is ever needed, the right tool is `common.requireRole(…, .operator)`, not a token-scope pre-check.

No code change was needed — `authorizeWorkspace` already implements broad semantics. Added a 10-line comment at the call site so a future reader sees the intentional widening and doesn't assume the M23_001 pre-check was dropped accidentally.

### 2. `getZombieWorkspaceId` → factored into `common.zig`

Lived in `integration_grants_workspace.zig` pre-M24. Now needed from `zombie_activity_api.zig` too. Three choices:

- Duplicate: silly, two copies of the same 10-line lookup.
- Export from grants-workspace: creates a dependency from `zombie_activity_api` → `integration_grants_workspace` which is backwards.
- Move to `common.zig`: natural, since `common` already owns `authorizeWorkspace`.

Moved to `common.zig`. Grants-workspace now calls `common.getZombieWorkspaceId`. Kept a one-line note in the grants file so git-blame readers find the rename.

### 3. IDOR test file → new file, NOT an extension of the steer test

Debated adding the IDOR tests directly to `zombie_steer_http_integration_test.zig` since that file already has TEST_TENANT_ID, TEST_WORKSPACE_ID, OTHER_WS_ID, JWKS, and the operator token wired up. Pragma argued for shared fixtures; pragma lost because:

- The file name lies if non-steer tests live in it.
- The existing `cleanupTestData` deletes by `workspace_id IN (TEST_WS, OTHER_WS)`, which wipes a shared-workspace test fixture. IDOR tests using a *different* `OTHER_WS_ID` avoid this.

New file `m24_001_cross_workspace_idor_test.zig` duplicates the server/token setup but uses a unique `OTHER_WS_ID` and zombie IDs so parallel runs with the steer test don't interfere. Scoped cleanup deletes only the IDs this file creates, never `TEST_WORKSPACE_ID` (shared with other files).

### 4. Connection reuse API → `queryByZombieOnConn` as the new primary, `queryByZombie` wraps it

`activity_stream.queryByZombie(pool, …)` acquired its own conn. The handler had already acquired one for `authorizeWorkspace` and kept it live under `defer pool.release`. Two conns held per request.

Fix: add `queryByZombieOnConn(conn, …)` that takes the caller's conn. Refactor `queryByZombie` to a 3-line wrapper that acquires and delegates. Handler uses the OnConn variant.

This is the same pattern as `killZombieOnConn`, `fetchZombieListOnConn`, `storeCredentialEncryptedOnConn`, etc. — M24 was moving toward single-conn handlers; `queryByZombie` was the holdout. Now captured as RULE CNX in `docs/greptile-learnings/RULES.md`.

### 5. Grant revoke SQL — workspace_id defence-in-depth

The existing `getZombieWorkspaceId + eql` check already rejects cross-workspace revokes at the app layer. The SQL scope is redundant *today*. Added per handoff anyway — matches the `killZombieOnConn` pattern and provides a backstop if a future refactor ever removes or bypasses the app-layer check. Integration tests still pass because the new WHERE clause is strictly more restrictive, not less.

## Trade-offs considered (and the path not taken)

- **Combining the stale-docstring fixes into a single commit vs per-file commits.** One commit is easier to review, per-file commits are easier to revert. Chose one commit — the changes are cosmetic and the blast radius of reverting them is zero.
- **Adding IDOR tests for `request_integration_grant` (zombie-identity auth).** Handoff item listed it but noted the in-code `caller.workspace_id == workspace_id` check already exists at `integration_grants.zig:129-132`. Skipped because a zombie-auth fixture is a separate apparatus (Session header or `zmb_` API key + `core.zombie_sessions` seeding) and the in-code guard is already verified by the existing grant integration tests. If a follow-up milestone adds zombie-auth fixtures, add the IDOR case then.
- **Touching `public/openapi.json`.** No endpoint signatures changed. OpenAPI stays as-is.
- **Opening a new `RULE WAUTH2`.** Considered. Decided that "workspace → zombie ownership" is a distinct constraint from "principal → workspace" and deserves its own letter code. Named RULE ZWO (Zombie-Workspace Ownership).
