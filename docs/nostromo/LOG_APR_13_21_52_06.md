## The use case in plain language

M14_004 measures whether a zombie's memory is actually making it smarter, or just accumulating junk. Four metrics per zombie:

1. **Recall-hit rate** — when the zombie looked something up, did it find it?
2. **Duplicate-action rate** — did it repeat itself (proof memory *didn't* help)?
3. **Memory size** — how much is it hoarding?
4. **Wrong-context-asserted rate** — did it confidently state something false from stale memory? Operator hits thumbs-down; that's the signal.

"Archetype" is the zombie's role template (lead_collector, ops_zombie, hiring_zombie, …). Aggregating per archetype lets the operator compare *all their lead_collectors* against *all their ops_zombies* in one view.

## Customer value — two archetype walkthroughs

### Lead Collector zombie (outbound to prospects)

- **Recall-hit low** → it's re-researching the same company every run instead of remembering "Acme already replied declined." Operator sees the number drop and knows the skill template regressed.
- **Duplicate-action high** → it pinged the same lead twice in a week. Difference between "persistent follow-up" and "spammed our ICP." Directly saves the customer's sender reputation and domain health.
- **Wrong-context** → zombie told a lead "per our last call…" when no call happened. One thumbs-down, and the rate surfacing at 3% of outreach tells the operator to tighten memory TTL before the next campaign.

### Ops zombie (on-call triage, incident response)

- **Recall-hit high is the whole point** → it should remember "this alert at 3am usually means the cache node flapped, runbook X." Low rate means it's paging humans for things it already solved last week — the zombie stopped earning its keep.
- **Duplicate-action** → restarted the same pod twice in 10 minutes. Memory isn't tracking its own actions. Real outage-maker, not a cosmetic issue.
- **Wrong-context** → asserts "I already restarted the DB" when it didn't. Catastrophic in ops. Thumbs-down is how the SRE flags it, and the rate tells the operator whether the zombie is safe to run unattended.

## Why this belongs in the release narrative

Without these metrics, memory is a black box — the customer pays storage and hopes it helps. With them, they can tell a good zombie from a confidently-wrong one *before* it burns a lead or an incident.

When this milestone ships, the changelog `<Update>` block should lead with this framing, not the metric names. The metric names are how; the archetype walkthroughs are why.
