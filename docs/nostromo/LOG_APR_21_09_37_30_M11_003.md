### Step 4 — signup_bootstrap facade + store

The transactional facade was straightforward once the schema was in place. Six inserts (`core.tenants`, `core.users`, `core.memberships`, `core.workspaces`, `billing.workspace_credit_state`, `billing.workspace_credit_audit`) inside one BEGIN/COMMIT. Fast-path replay check on `oidc_subject` before opening the tx — hit means `created:false` with zero writes.

An interesting owner question surfaced here: "Can these be done in a single query or in one batch transaction commit, so they all get done or gets rolled back?" Both answers are yes. Technically they could be one CTE-chained statement, but that would make the Heroku name collision-retry loop (which needs conditional re-execution based on the previous INSERT's rowcount) very awkward. Six discrete statements inside one transaction gives the same atomicity guarantee without contorting the retry logic.

That detail came back to bite me later (see the tier-3 section below).

### Step 12 — tier-3 fresh DB (and the hang that taught me pg-driver state semantics)

First `make test-integration` run hung for 20+ minutes. `pg_stat_activity` showed a pg backend in `idle in transaction` on my `tryInsertWorkspace` SELECT, holding the connection hostage. The test binary was sleeping at 0% CPU. Owner granted the kill permission I'd asked for.

Root cause of the hang was a cursor-lifetime mismatch. The original `tryInsertWorkspace` used a two-roundtrip SELECT-then-INSERT pattern. The `PgQuery` wrapper's `defer deinit()` auto-drains, but the block-scoped `defer` inside a `blk: { ... }` expression was interacting badly with the enclosing transaction state on retry. Collapsed the logic to a single `INSERT ... ON CONFLICT (tenant_id, name) WHERE name IS NOT NULL DO NOTHING` against the partial unique index. `conn.exec` returns the rowcount; `> 0` means the name landed. One statement, no cursor, handles concurrent-race cleanly.

That fix cleared the hang, but surfaced a second latent bug. The next tier-3 run hit test Dim 3.4 (transactional rollback) and the test-after query failed with `error.PG`. Turned out my `errdefer conn.exec("ROLLBACK", .{}) catch {};` pattern was wrong in a specific way: after a unique-violation, the pg driver's connection is in `.fail` state. `conn.exec` goes through `execOpts` which short-circuits in that state — ROLLBACK never actually reaches the wire, and the session stays stuck in an aborted tx. The pg driver exposes `conn.rollback()` for exactly this case — it uses `execIgnoringState` and flushes to `ReadyForQuery` regardless of state. Switched to that, tier-3 went green: `✓ [zombied] Full integration suite passed`.

Commit: `29c904b8 fix(m11-003): single-statement workspace upsert + proper rollback on fail`

Worth a RULE entry someday: "use `conn.rollback()` not `conn.exec(\"ROLLBACK\"…)` in errdefer paths, because the driver's state machine swallows ROLLBACK in fail state." Filing as a greptile-learnings candidate for a follow-up session; didn't want to widen the scope of this CHORE(close).

- **Tenant name derivation:** email local-part, fallback `"personal"` if empty. Owner confirmed mid-session.
- **Membership role:** literal string `"owner"` — tenant-scoped, covers default workspace + any workspaces the user creates later in their personal account. Per-workspace roles are a team-accounts milestone, out of scope.
- **Workspace name nullability:** nullable + partial unique index. Making `name` NOT NULL would have cascaded into 11 fixture inserts + 3 production handlers that don't supply a name; not worth the tour.
- **`core.tenants.api_key_hash` fate:** deleted entirely (14 file edits), not sentinel-stuffed, because nothing reads it — `core.api_keys` (M28_002) is the live auth table.
- **OpenAPI source of truth:** the YAML splits, not the bundled `openapi.json`. Direct edits to the bundled file would get overwritten on next `make openapi`.
- **Heroku name combinatorial space:** 32 × 32 × 1000 = 1,024,000. Owner asked twice; math is the same both times. Practical per-tenant collision probability at signup is zero; the 8-attempt retry loop is a guard against a buggy generator.

## Rules considered for addition (greptile-learnings)

Not landing them in this session per scope, but flagged for a follow-up:

- **Always use `conn.rollback()` in `errdefer`, not `conn.exec("ROLLBACK", .{})`** — the driver's state machine swallows ROLLBACK when the connection is in fail state, leaving the session stuck in aborted tx. This caused ~40 minutes of debugging on tier-3. A grep for `conn.exec("ROLLBACK"` returns 5 hits in the codebase; all should be audited.

- **Prefer `INSERT ... ON CONFLICT ... DO NOTHING` + rowcount over `SELECT ...; if not exists INSERT` inside transactions.** Single statement, no cursor lifetime to reason about, handles concurrent writers, and sidesteps the kind of mid-tx cursor wedging I just hit.
