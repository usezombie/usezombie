## What we established (not obvious, worth preserving)

### The observability/control split

M12 + M13 + M14 together cover the **observability half** of the operator experience:
- Activity stream (what the zombie did)
- Firewall metrics (what it proxied, what was blocked)
- Credential usage log (which key, which zombie, when)
- Memory entries (what the zombie knows)
- Kill switch, trust score, spend tracker

They cover **none of the control half**:
- Creating a zombie from the dashboard
- Configuring a trigger (webhook URL, cron schedule)
- Managing integration grants
- Approving pending gate actions
- Configuring the LLM provider (BYOK)

An operator who only uses the dashboard can **observe** their zombie but cannot **set it up, unblock it, or provision it**. The dashboard as designed by M12–M14 leaves every setup step in the CLI.

### Approval inbox is the single most impactful gap

All three reference archetypes show approval gates as core mechanics:
- Hiring Agent: offer letters require per-action approval
- Lead Collector: hot leads (score ≥ 90) require approval before acting
- Ops Zombie: paging on-call, auto-scale always require approval

Approvals currently flow through Slack DMs only. If a zombie stalls waiting for an offer-letter approval, the M12 dashboard shows the zombie as "Active" — no hint of the pending gate. A CTO who opens the dashboard to check on the hiring agent has zero path to act. This makes the dashboard give false confidence rather than accurate status.

### UZ-GRANT-001 is the most common failure mode; dashboard has no grants surface

Every zombie integration guide lists `UZ-GRANT-001` (no approved grant) and `UZ-GRANT-002` (grant pending) as the top troubleshooting items. There is no grants panel anywhere in M12–M14. An operator whose zombie stopped calling the HubSpot API can see the `UZ-GRANT-001` error in the activity stream but cannot diagnose or fix it from the dashboard.

### BYOK is not in any spec

M15_001 (credit metering, done) tracks usage. But BYOK (bring your own Anthropic/OpenAI key) has no spec. The user model is:
- Sign up → get free credits (hosted Claude, X runs)
- Use product → credits deplete
- Path A: upgrade UseZombie plan (they manage the provider)
- Path B: BYOK — add your own `ANTHROPIC_API_KEY` to the vault, zombies run on your quota

Provider credentials are distinct from tool credentials:
- Tool credentials: stored in vault, injected into outbound HTTP via firewall (M5_001 / M13)
- Provider credentials: used by UseZombie runtime itself to call LLM APIs — never injected into zombie tool calls, never visible to the zombie

This needs a separate UI surface (Settings > Provider), a separate CLI command (`zombiectl provider set`), and a new credential type in the vault.

### The invite code model is UseZombie using UseZombie

The operator added a new constraint: users sign up via invite code. The invite code is generated and sent by the Lead Nurturer Zombie — UseZombie's own instance of itself running against its own lead pipeline. When a qualified lead comes in, the zombie:
1. Scores the lead
2. Calls `POST /v1/invites` to generate an invite code (UseZombie internal API, gated by approval)
3. Sends the invite code to the lead via email (or Slack, or wherever they came in)
4. Lead enters code at `app.usezombie.com/signup` → account created, credits loaded instantly

This must work in CLI (`zombiectl auth signup --invite-code IVT-xxxx`), UI (invite code entry on signup page), and API (`POST /v1/auth/signup { invite_code: "..." }`). An agent/pipeline can receive and redeem invite codes programmatically — important for enterprise customers automating workspace provisioning.

## Two DX flows mapped end-to-end

### Lead Collector: CLI vs UI

**Day 1 setup (CLI — works today):**
1. `zombiectl login`
2. `zombiectl workspace create --name acme-prod`
3. `zombiectl credential add agentmail / hubspot / slack`
4. `zombiectl zombie create --name "lead-collector" --skill lead-collector-v1`
5. `zombiectl zombie triggers list --zombie zom_01` → copy webhook URL to AgentMail
6. `zombiectl grant request --zombie zom_01 --service hubspot --reason "..."` → approve in Slack DM
7. `zombiectl zombie firewall set --zombie zom_01 --config '{...}'`
8. Send test email → `zombiectl runs replay --run run_01`

**Day 1 setup (UI — after M12–M14 + new milestones):**
1. `app.usezombie.com/signup` → enter invite code → credits loaded (M25)
2. Onboarding wizard: pick "Lead Collector" archetype (M23)
3. Wizard step 2: add agentmail/hubspot/slack credentials (M13)
4. Wizard step 3: copy webhook URL from trigger panel (M19)
5. Wizard step 4: request grants — reason pre-filled (M20)
6. Wizard step 5: approval gate pre-configured per archetype (M19)
7. Wizard step 6: provider — hosted credits or BYOK (M22)
8. Launch → activity stream shows first run (M12)
9. Approve hot leads from approval inbox (M21)

### Blog Writer + Poster: CLI vs UI

**Day 1 setup (CLI):**
1. `zombiectl credential add serper / ghost_admin / unsplash`
2. `zombiectl zombie create --name "blog-writer" --skill blog-writer-v1`
3. `zombiectl zombie schedule --zombie zom_blog --cron "0 9 * * 2"`
4. `zombiectl grant request --zombie zom_blog --service ghost_admin`
5. `zombiectl zombie firewall set --zombie zom_blog --config '{ "endpoint_rules": [{ "domain": "ghost.io", ..., "action": "requires_approval" }] }'`
6. Seed memory with topic backlog + style prefs via `zombiectl memory import`

**Ongoing (UI gap — Day 1 stalls here without M19–M25):**

- Zombie drafts a blog post → hits approval gate → Slack DM arrives.
- Operator opens dashboard instead. Sees: zombie "Active". No indication it's been waiting 40 minutes for sign-off.
- This is the exact moment the dashboard fails this use case.
- M21 (Approval Inbox) fixes this: badge shows "1 pending", inbox shows the draft post preview.

## End-user = human OR agent: the two paths that matter

Every milestone must satisfy both:

**Human operator path:** dashboard or CLI. Does not write code. Opens app.usezombie.com, reads the state, clicks Approve, or runs `zombiectl` commands from a terminal. Cares about: is it working, what did it do, I need to approve this now.

**Agent/pipeline path:** API with `zmb_` or workspace token. Does not use the dashboard at all. Provisions workspaces, creates zombies, configures triggers, redeems invite codes, programmatically approves gates (for CI/CD pipelines that have their own orchestration). Cares about: POST endpoints, response schemas, idempotency.

Every new milestone spec includes both paths explicitly. No milestone may be designed for human-only use.

## Decisions recorded

1. BYOK credentials are a distinct type from tool credentials — they are used by the UseZombie runtime, not injected into outbound zombie HTTP calls. Vault stores them but a new `credential_type = llm_provider` field distinguishes them at the API layer.

2. The invite code is generated by the Lead Nurturer Zombie via `POST /v1/invites` (UseZombie internal API). This call is gated by an approval if the invite value is significant. Below a threshold (e.g., small-credit free invites) it runs autonomously.

3. Approval gates in M4_001 are per-action. M21 adds the UI over them. The gate state and gate resolution API are purely M4_001 plumbing; M21 is presentation only. No M4 changes needed.

4. M12's settings page (§5.0) will be extended by M22 to add a "Provider" tab. M12's scope stays as-is; M22 adds the tab.

5. The onboarding wizard (M23) is B8 — it depends on the underlying CRUD being stable (M19, M20, M22). Shipping the wizard before the primitives would mean the wizard breaks every time a primitive changes.
