## What I set out to do

## What the spec actually said when I read it

Three factual errors against repo state:

1. Claimed today's `install` POSTs to the server. It does not — it's a **local scaffolder** that copies `zombiectl/templates/<name>/*.md` into cwd. The POST is `zombie up`.
2. Named the file to edit as `zombiectl/src/commands/zombie_install.js`. No such file — install logic lives in `zombiectl/src/commands/zombie.js` alongside `commandUp`, `commandStatus`, etc.
3. Cited `samples/platform-ops/` as the smoke target. That directory doesn't exist yet (M37_001 ships it). Tests and goal block had to use an existing sample — `samples/homelab/`.

Also stale: a cross-reference to "M33_001 owns the `--template` flag" that didn't match repo reality (M33_001 is Worker Control Stream + Chat, not templates).

## The conversation that reframed the workstream

User asked why `install` should print the webhook URL. That pulled on a thread: webhook URLs are only coherent when the zombie is registered server-side. Today, `install` (scaffold) doesn't register. `up` does. So "install prints webhook" is only true if install is doing the POST.

User confirmed the architecture doc's guarantee: **the zombie is live on create** — per `ARCHITECTURE_ZOMBIE_EVENT_FLOW.md §3, §7, §8-inv-#1`, `POST /v1/workspaces/{ws}/zombies` INSERT + XGROUP CREATE + XADD `zombie:control` are atomic, and the watcher spawns the zombie thread before the 201 ACKs. No trigger-to-start step.

That collapsed the workstream scope:

- If install-from-path POSTs and the zombie is live before the command returns, then `up` is redundant for this path.
- User's call: "1 command `zombiectl install`, remove `zombiectl up`, only `--from`."
- And: "just print the zombie id" — webhook URL is available via `zombiectl status`/`list` when external wiring needs it. JSON mode still includes webhook for scripting parity.

So M19_003 became an **install-surface rewrite**, not a flag add. Pre-v2.0 alpha — removal is clean, no deprecation shims.
