## Fixing the TOCTOU — and what `zombie_api.zig:115` got wrong

The natural fix is: drop the pre-flight SELECT entirely; let the DB's unique constraint arbitrate; map SQLSTATE `23505` (unique_violation) on INSERT failure to `error.NameTaken`. Before applying it, I grepped for prior SQLSTATE-inspection patterns in the repo and found `src/http/handlers/zombie_api.zig:115`:

```zig
fn isUniqueViolation(_: anyerror) bool {
    // pg.Pool returns error.PGError for all Postgres errors (connection, constraint, cast).
    // We cannot distinguish unique_violation (SQLSTATE 23505) from other PGErrors
    // because pg.Pool does not expose structured SQLSTATE codes.
    // Return false to let the caller surface a 500 instead of a misleading 409.
    return false;
}
```

This claim is **outdated**. I confirmed against the pinned pg.zig source (commit `a4351154a00488d26740b4378872be03bd562d27`, which main bumped to in PR #223):

- `pg.Conn` has a public `err: ?proto.Error` field (`src/conn.zig:28`).
- The pg test at `src/conn.zig:660` reads `c.err.?.code` and asserts it equals `"42601"` — SQLSTATE codes are in fact exposed structurally as `conn.err.?.code`.

So the fix landed as:

```zig
_ = conn.exec(
    \\INSERT INTO core.api_keys (id, tenant_id, key_name, description, key_hash, created_by, active)
    \\VALUES ($1::uuid, $2::uuid, $3, $4, $5, $6, TRUE)
, .{ id, tenant_id, body.key_name, description, key_hash[0..], user_id }) catch {
    if (conn.err) |pe| {
        if (std.mem.eql(u8, pe.code, "23505")) return error.NameTaken;
    }
    return error.DbError;
};
```

I deferred updating `zombie_api.zig:115` out of respect for scope discipline — that file is not in M28_002's blast radius and the `isUniqueViolation` shim affects zombie creation, not API key creation. Capturing this as a Greptile Learning candidate: the "`pg.Pool` does not expose SQLSTATE codes" comment is wrong against the current pinned commit. A follow-up workstream (probably scoped to `zombie_api.zig` alone) should delete `isUniqueViolation` and inline the same `conn.err.?.code == "23505"` check there, then upgrade `ERR_ZOMBIE_NAME_EXISTS` from never-fires to an actual 409 path. Not doing it now to avoid scope creep on this PR.

After the TOCTOU fix, I re-ran the integration suite against the same fresh DB — still green (exit 0). Dimension 3.5 (duplicate `key_name` → 409) passed without the pre-flight SELECT, confirming the `23505` mapping works end-to-end.
