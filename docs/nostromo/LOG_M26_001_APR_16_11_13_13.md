## Two scope surprises that moved the plan

### Surprise 1 — `request_id` → `requestId` was a bad rename

The audit agent flagged "spec says camelCase for acronyms (RAD §1)" and
proposed renaming `request_id` → `requestId` in memory responses. I accepted
that finding in the initial spec and executed it. User interrupted mid-flight:
"Can i know this why are we doing camelCase, is that the norm?"

They were right. Re-reading RAD §1 carefully: the rule **"Use lower camelCase
for acronyms (`nextUrl`, not `nextURL`)"** governs **how acronyms are spelled
inside an identifier** — write `Url` not `URL`. Every other example in the same
doc is snake_case (`created_at`, `product_id`, `expiration_days`,
`phone_number`). The whole codebase is consistently snake_case on the wire.
The rename was breaking-change theater against a convention that didn't
actually exist.

Reverted the rename. Saved a feedback memory so future me doesn't make the
same mistake:
`~/.claude/projects/-Users-kishore-Projects-usezombie/memory/feedback_acronym_rule_vs_casing.md`.
The audit agent's output is a signal, not a verdict — especially for convention
claims, grep the codebase before accepting.

### Surprise 2 — §6 auth extraction was already done

Spec §6 called for moving `authorizeWorkspace`, `getZombieWorkspaceId`,
`authorizeWorkspaceAndSetTenantContext` from `src/http/handlers/common.zig` to
a new `src/auth/rbac/workspace.zig`. Before executing, I grepped `src/auth/`
for `@import("pg")` — zero hits. `build.zig:184` has a `test-auth` target that
imports **only** httpz. M18_002 already finished the real extraction.

The three functions I planned to move are all DB-coupled — they take
`*pg.Conn` as their first argument. Moving them into `src/auth/` would either
(a) force pg into `test-auth`'s import list, breaking the exact portability
contract the user wanted preserved, or (b) require a `WorkspaceLookup`
dependency-injection refactor that's a milestone of its own.

The honest call: **"ensure auth is built in this PR" is already satisfied**.
`zig build test-auth` is green on this branch; no action needed. Marked §6
ALREADY-SATISFIED in the spec with a detailed rationale so the next reader
doesn't try again without resolving the DI question first.

This is the more important lesson of the two: **spec says "move this
function" is NOT the same as "this function can be moved."** Always check
the dependency closure before committing to an extraction spec.

## For the next agent

Consider adding a `check-openapi-route-coverage` lint gate — every `Route`
enum variant must have a matching `openapi.json` path+method. We just closed
a 26-operation drift gap; a structural check prevents it from silently
regrowing as new routes are added. The script is ~30 lines of Python:
parse `src/http/router.zig` for the `Route` union variants, parse
`public/openapi.json` for paths × methods, assert 1:1 coverage. Hook it
into `make lint` alongside the ErrorBody check.

Some of the openapi entries I added have minimal response schemas (just
`{ type: 'object', description: '...' }`) where the actual response shape
is either relay-specific (`spec/template`, `spec/preview`), operator-
internal (`internal/v1/telemetry`), or tool-dependent (`/v1/execute`
returns whatever the invoked tool returns). Filling those in with concrete
types is a polish pass, not a coverage gap — they don't block SDK
generation, they just narrow the generated types.
