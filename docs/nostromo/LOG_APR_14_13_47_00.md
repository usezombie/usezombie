## What happened

Started M23_001 as specified — the full "memory bridge" spec (durable steering that persists across runs, LLM scope inference, steering_bridge module). Before writing any code, asked a clarifying product question: do you want corrections to persist across runs, or only affect the current run?

**User confirmed scope: live chat steering only.** No memory persistence. M14_001 is parked.

Rewrote the spec on the spot before implementation. The new scope is:
- `POST /v1/zombies/{id}:steer` → find active `execution_id` in `core.zombie_sessions` → write to Redis `run:{execution_id}:interrupt` (300s TTL) → existing worker gate loop picks it up.

## Key decisions

**Dropped from original spec:**
- `src/memory/zombie_memory.zig` — no memory writes
- `src/memory/steering_bridge.zig` — no bridge/scope inference
- `src/http/events/sse_events.zig` — no SSE event emission
- `src/http/handlers/run_interrupt_http.zig` — no v1 runs alias (M10_001 removed runs routes; no business reason to re-add without memory bridge)
- Worker interrupt polling logging changes — deferred

**Route ordering matters:** `zombie_steer` match (`/v1/zombies/{id}:steer`) must come before `matchZombieId` in router.zig `match()`. Without this, a path like `/v1/zombies/abc123:steer` would fall through to the plain-id matcher and return wrong route. Added comment in router.

**Redis write failures are non-fatal:** If Redis is down, steer returns 200 with `run_steered: false` and a warn log. Operator sees it didn't land but nothing crashes.

**`resolveActiveExecution` returns `??[]const u8`** (optional optional) — outer null means "db error, response written"; inner null means "no active session, valid state". This keeps the success path tight without a separate error flag.
