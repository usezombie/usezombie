## What I did

### 1. `src/auth/middleware/svix_signature.zig` (new, 168 lines)

Comptime-generic on `LookupCtx` to mirror the `webhook_sig` pattern (RULE NTE — no `*anyopaque` in the lookup callback). Key design decisions:

- **Base64 flexibility.** Svix secrets and signatures in the wild arrive both padded (`standard`) and unpadded (`standard_no_pad`). The prior handoff claimed Svix uses no-pad; inspection of the Svix Rust and Go libraries shows they use `StdEncoding` (padded). To be safe against either form, `decodeBase64` tries `standard.Decoder` first and falls back to `standard_no_pad.Decoder` — one function, one fallback, no ambiguity.
- **Base64Decoder is a value, not a type.** First attempt declared `tryDecode` with `comptime Decoder: type` — Zig 0.15 rejected it because `std.base64.standard.Decoder` is a `Base64Decoder` instance. Fixed by passing a `*const Base64Decoder` by pointer.
- **Injectable `now_seconds`.** Mirrored `slack_signature.zig`'s `NowSecondsFn` pattern so tests can run with a fixed clock. The canonical `hs.isTimestampFresh` reads `std.time.timestamp()` — fine for production, but flaky when a test needs "exactly 50s before now." Added a local `isTimestampFresh(ts, now, max_drift)` variant that takes `now` as a parameter. Exported `pub` so tests can pin the boundary at ±max_drift.
- **Single `execute` method at 14 lines.** All the actual verification logic lives in free functions (`verify`, `anySignatureMatches`, `decodeBase64`, `tryDecode`) so no method exceeds the 50-line cap.
- **Misconfig path logs before failing.** Missing `whsec_` prefix: `log.warn` + 401. Not an attack, but operators need to see it. All tamper/tamper-adjacent failures go straight to 401 without log noise.

## Dead end / recovered

**`std.base64.standard.Decoder` as a type.** First `svix_signature.zig` compile failed with:

```
error: expected type 'type', found 'base64.Base64Decoder'
    if (tryDecode(std.base64.standard.Decoder, dest, src)) |n| ...
```

`std.base64.standard.Decoder` is a `Base64Decoder` **value**, not a generic type. Fixed by passing `*const std.base64.Base64Decoder` by pointer instead of a comptime `type` parameter. 30-second fix once the error was understood; logged here because the naming convention misled me into thinking `.Decoder` was a namespace.

## Decision: defer §3.8 + §5.8 to M28_003

The handoff called for full HTTP integration tests: `receive_linear_webhook_e2e` and `receive_svix_webhook_e2e`. I walked the existing HTTP integration tests (`byok_http_integration_test.zig` 524 lines, `rbac_http_integration_test.zig` 430 lines) and found each stands up its own `pg.Pool` + Redis + JWT verifier + middleware registry + in-process HTTP server. - `webhook_sig_test.zig` — 3 HMAC strategy 2 tests covering §3.3–3.5.

- `lookupSvix` reuses `parseSignature` + `crypto_store.load` — the exact vault path already under integration coverage via §3.6.

The marginal signal from a full HTTP integration test is a smoke check against wiring that's already proven one layer down. I took the call to defer §3.8 + §5.8 to a new spec:

**`docs/v2/pending/P2_API_M28_003_WEBHOOK_E2E_INTEGRATION_TESTS.md`**

Priority P2 — smoke coverage for shipped behavior, no user-visible change. Surfaced the tradeoff to the user; they approved the defer.

This keeps M28_001's diff focused. If the scaffolding cost of integration tests is a recurring problem (it already affects 6+ workstreams), the right move is a shared `WebhookTestServer` helper — captured in M28_003's Out of Scope as a note for future consolidation.
