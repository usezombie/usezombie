## What I *didn't* ship, and why

Originally M12 was scoped to deliver both the backend endpoints *and* the full dashboard pages (overview, zombies list, zombie detail, firewall placeholder, settings). I stopped mid-EXECUTE (after landing the shared UI primitives but before starting on pages) and narrowed the scope.

### The surfacing moment

While standardizing the design tokens in `globals.css`, I audited what the existing `@usezombie/design-system` Button actually produces. Three things fell out:

1. **The design-system package is not self-contained.** Its `Button.tsx` emits `.z-btn` / `.z-btn--ghost` / `.z-btn--double` class names, but the actual `.z-btn { ... }` CSS rules are defined in `ui/packages/website/src/styles.css` — the *consumer*, not the package. Dropping this Button into the Next.js app renders unstyled text. I verified with a `grep -rln "\.z-btn" --include="*.css"` — only the website's stylesheet defines them.

2. **The design-system package is not RSC-safe.** `design-system/Button.tsx` has `import { Link } from "react-router-dom"` at the top. Next.js 16 App Router's Server Components reject that import entirely; even using the Button in a Server Component would force the whole subtree client-side.

3. **The app had its own parallel Button.** `ui/packages/app/components/ui/button.tsx` is a shadcn-style clone I refactored as part of this milestone's token work. It doesn't consume `@usezombie/design-system/Button` (it can't, per 1 + 2) and has drifted visually in small ways even though both share `--z-orange` as the brand accent.

### The decision

The user surfaced this explicitly: *"I prefer that we ship the Design system M26 first, that is important than finishing up the work. And we could move the others to do in this in the next milestone."*

Building the dashboard pages against the app-local Button — then rewriting every page to consume the unified Button after M26 lands — means writing the pages twice. Nobody wins. Better to ship the foundation that's actually done (backend + primitives + tokens), unify the design system next, then build the pages once.

## Decisions made that don't belong in commit messages

### Why per-zombie billing is at `/zombies/{id}/billing/summary`, not `/billing/summary?zombie_id=`

The user questioned my initial draft that had `?group_by=zombie_id`. Per `docs/REST_API_DESIGN_GUIDELINES.md` §2 (reflect hierarchy in path) and §5 (resource IDs in path, not query), the zombie is a sub-resource of the workspace and its billing slice should be nested under it. SDK ergonomics favor this too: `workspaces.billing.summary(ws)` vs `workspaces.zombies.billing.summary(ws, z)` are symmetric calls. The response envelope is identical between the two, so SDK codegen emits a single `BillingSummary` type consumed by both scopes. Documented inline in `zombie_billing_summary.zig` and in the openapi entry.

### Why the workspace billing summary was a zero-stub before M12

M10_001 dropped `billing.usage_ledger` and `scoring.agent_run_scores` tables. `workspaces_billing_summary.zig` was left returning zeros with a comment saying "M15_001 will wire zombie credit metering." M15 did land but didn't wire the summary. My M12 handler wouldn't have made sense returning real data for one scope and zero for the other, so I wired both through the new shared `billing_summary_store` module. Side effect, not the feature — documented in the commit message and the spec §2.3.

### Why `:stop` returns 409 on re-call instead of idempotent 200

The original spec said "Returns 409 if already terminal." I considered arguing for idempotent 200 (RULE IDMP-adjacent), but RULE IDMP is scoped to INSERT guards for resources with UNIQUE constraints, not UPDATE idempotence. For a destructive-ish operation like "stop this zombie," a distinct 409 with `UZ-ZMB-010` gives the operator UI a chance to say "this is already stopped" rather than silently re-confirming the no-op. The kill-switch UI will render this as a toast with the error code surfaced for debugging. Matches the existing 409 pattern on `POST /zombies` when a zombie with the same name exists (`UZ-ZMB-006`).

### Why the integration test file is 358 lines (over the 350 gate)

The filename matches the `_test.` exempt pattern per RULES.md's FLL carve-out. Test files carry fixtures + seed + assertions and forcing a split harms readability. The gate explicitly allows this. My two production handler files (`zombie_lifecycle.zig` at 164 and `zombie_billing_summary.zig` at 127) are well under cap.

### Why I didn't use `asChild` on StatusCard

I tried during the initial primitive write, got pushed on it by ARIA concerns, and discovered that Radix `Slot`'s single-child clone model is incompatible with a component that renders its own internal markup (`<dl><dt><dd>`). Slot clones the first child and merges props — when I wrapped the card in `<Slot>` and passed `<a>` as the consumer's child, Slot ignored the `<a>` and instead merged card classes onto the `<dl>`. The pattern works for leaf primitives (Button, Badge) where the whole render is `{children}`. For compositions like StatusCard, the right pattern is to wrap the whole card in `<Link>` externally. Documented in the source file's header comment, and noted as an explicit test case: `it("does not accept asChild ...")`.

### Why `components.json` uses `baseColor: "zinc"`

Closest shadcn base palette to the dashboard's dark-first brand. It's not actually consumed (we override every semantic token via `globals.css`), but shadcn CLI requires one of its standard bases. `zinc` vs `slate` vs `stone` are all cosmetically close given our overrides; zinc is what assistant-ui uses.

### Why the website's Button exists at all if we have shadcn

Historical. The website (Vite SPA) predates shadcn adoption in this project. It's built against `@usezombie/design-system`'s BEM-style components with CSS rules in its own `styles.css`. The app (Next.js 16 App Router) came later and adopted the shadcn paradigm because it's RSC-compatible and framework-agnostic via Radix Slot. M26 unifies by migrating the website onto the shadcn paradigm — not the other way around.
