### Step 4 — `src/state/signup_bootstrap.zig` + `src/state/signup_bootstrap_store.zig`

**Structural divergence from the m11 worktree draft:** the prior draft referenced `workspace_providers` and `attachAllPlatformDefaults`. **Those don't exist on main and are being deliberately left out** — platform LLM key resolution is runtime-only (worker reads `core.platform_llm_keys` + admin workspace vault at LLM-call time, not signup-time). **Do not port that code path.**

**What to build:**

`signup_bootstrap.zig` — the transactional facade:

```zig
pub const BootstrapParams = struct {
    oidc_subject: []const u8,
    email: []const u8,
    display_name: ?[]const u8 = null,
};

pub const Bootstrap = struct {
    user_id: []u8,
    tenant_id: []u8,
    workspace_id: []u8,
    workspace_name: []u8,
    created: bool,

    pub fn deinit(self: *Bootstrap, alloc: std.mem.Allocator) void;
};

pub fn bootstrapPersonalAccount(
    conn: *pg.Conn,
    alloc: std.mem.Allocator,
    params: BootstrapParams,
) !Bootstrap;
```

Inside:
1. Fast-path replay check: `SELECT user_id, tenant_id, workspace_id, workspace_name FROM core.users u JOIN core.workspaces w ON w.tenant_id = u.tenant_id WHERE u.oidc_subject = $1 LIMIT 1`. Hit → return existing rows with `created = false`. (Actual query shape depends on how workspace↔user↔tenant is joined — verify against `schema/001_core_foundation.sql` first.)
2. Open transaction: `conn.exec("BEGIN", .{})`.
3. `INSERT INTO core.tenants (…) VALUES (…) RETURNING tenant_id`. Tenant name: email local-part (everything before `@`), falling back to `"personal"` if empty.
4. `INSERT INTO core.users (oidc_subject, email, display_name, …) VALUES (…)`.
5. `INSERT INTO core.memberships (tenant_id, user_id, role) VALUES (…, 'owner')`.
6. Pick a Heroku name with collision retry (up to 8 attempts): `INSERT INTO core.workspaces (tenant_id, name, …) VALUES (…) ON CONFLICT DO NOTHING` → retry if rowcount 0.
7. `INSERT INTO billing.workspace_credit_state (workspace_id, remaining_cents, …) VALUES (…, 0, …)`.
8. `INSERT INTO billing.workspace_credit_audit (…, source='signup_bootstrap', …)`.
9. `COMMIT` on success, `ROLLBACK` on any error (`errdefer` pattern).

Use `conn.exec()` not `conn.query()` for inserts (no rows needed back except the RETURNING cases — for those use `conn.query().drain()` per RULE FLS).

`signup_bootstrap_store.zig` — split out the single-row helpers: `findExistingByOidcSubject`, `insertTenant`, `insertUser`, `insertMembership`, `tryInsertWorkspace` (returns bool for collision retry), `insertCreditState`, `insertCreditAudit`. Keeps `signup_bootstrap.zig` under 350 lines.

**Tenant kind:** `core.tenants` has no `tenant_kind` column on main. Do not invent one. The plan's prior draft's reference to `kind = 'personal'` was from the now-abandoned branch — ignore.

**`api_key_hash` sentinel:** Check whether `core.tenants.api_key_hash` is NOT NULL in the current schema. If yes, use a sentinel like `"oidc_bootstrapped"` (the abandoned draft's pattern). If the column has been dropped or made nullable in a later M28 migration, leave NULL.

**Priors:** `src/state/signup_bootstrap*.zig` in the abandoned `usezombie-m11-invite-signup` worktree. Reference-only — adapt heavily to main's schema reality.

## Risks / gotchas

- **`core.tenants` schema drift.** The abandoned M11 draft referenced `tenant_kind`, `api_key_hash` sentinel, `name`. Check current schema at step 4: `grep -n "CREATE TABLE core.tenants\|ALTER TABLE core.tenants" schema/*.sql`. Do not guess — read the actual columns. M28's migrations may have reshaped things.
- **`billing.workspace_credit_state` and `workspace_credit_audit` shapes.** Same drill — read the schema at step 4. Columns: `workspace_id`, `remaining_cents`, `currency`, `consumed_cents`, etc. Exact names matter for the INSERT.
- **Env var read in handler.** `CLERK_WEBHOOK_SECRET` via `std.process.getEnvVarOwned(std.heap.page_allocator, "CLERK_WEBHOOK_SECRET")`. The Slack middleware precedent (`src/http/handlers/slack_events.zig:155`) does this with `defer page_allocator.free(secret)`. Copy that pattern exactly.
- **Missing env var.** Empty or missing `CLERK_WEBHOOK_SECRET` → 500 `UZ-INTERNAL-*`, **not** 401. Otherwise attackers learn "this endpoint has no secret configured."
- **Idempotent replay check.** If two Clerk webhooks for the same user.created arrive within the same second (extremely rare but Clerk does retry), the transaction's unique constraint on `core.users.oidc_subject` will catch the race — one commits, one gets a constraint error and rolls back. Next replay takes the fast path. No special race handling needed beyond the errdefer ROLLBACK.
- **Spec path changes.** Spec file moved `pending/P1_UI_CLI_API_M11_003_INVITE_SIGNUP_ONBOARDING.md` → `active/P1_API_M11_003_CLERK_SIGNUP_BOOTSTRAP.md`.
