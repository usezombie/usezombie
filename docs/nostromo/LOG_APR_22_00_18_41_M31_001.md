## §0.2 — OTLP histogram export

Problem was at `otel_export.zig:81-84` — the converter explicitly skipped `_bucket{`, `_sum `, `_count ` lines when translating Prometheus text to OTLP JSON. Histograms never reached an OTLP collector, and the test at line 260 *asserted* that skipping.

Prometheus histogram blocks span multiple lines, so I couldn't just add a case to the single-pass scalar loop. Extracted a new helper file `otel_histogram.zig` (201 lines) with:
- `Bucket { upper_bound, cumulative_count }` — `+Inf` stored as `std.math.inf(f64)`.
- `Histogram { name, buckets, sum, count }` — owns its name allocation.
- `Accumulator` — `StringHashMap(*Histogram)` keyed by base name; `tryIngest(line)` returns true if the line was consumed.
- `writeOtlp(writer, now_ns, first_ptr)` — sorts buckets, converts **cumulative → delta** bucket counts (Prometheus is cumulative, OTLP is non-cumulative), emits `explicitBounds` (excluding `+Inf`) and `bucketCounts` (len = bounds + 1) with `aggregationTemporality: 2` (CUMULATIVE).

In `otel_export.zig`: added a `histogram` case to the `# TYPE` parser, routed all histogram body lines through `acc.tryIngest`, flushed via `acc.writeOtlp` after the scalar loop. Flipped the old negative test to assert the three histogram field names **do** appear in OTLP JSON alongside the correct OTLP shape, and confirmed the raw Prometheus helper-series names (`_bucket`, `_sum`, `_count`) do NOT appear (they shouldn't — we emit OTLP fields, not Prom names). Added three unit tests in `otel_histogram.zig` covering ingest, non-match, and delta conversion.

Why split out `otel_histogram.zig` instead of inlining: estimated ~80-100 lines added to `otel_export.zig` would push it over the 350-line gate. File sizes now: otel_export.zig 292, otel_histogram.zig 201.

## §0.3 — per-(workspace, zombie) label

Merged into the §0.1 edit because it touched the same file. Rewrote `metrics_workspace.zig` to key slots on composite `(ws_id, zombie_id)` rather than `ws_id` alone:

- `Slot` now holds both `ws_id` and `zombie_id` byte buffers (48 bytes each, matching existing `WS_ID_LEN`).
- `compositeHash(ws, zombie)` uses Wyhash with a `\x00` separator between fields to avoid `"a"+"bc"` colliding with `"ab"+"c"`.
- Signature changed: `addTokens(ws_id, zombie_id, tokens)`. No old callers to migrate — the helper was dormant.
- Render line: `zombie_agent_tokens_by_workspace_total{workspace_id="W",zombie_id="Z"} N`.
- `MAX_WORKSPACES` renamed to `MAX_SLOTS` (= 4096) to reflect that it now counts composite pairs, not workspaces.
- Overflow label pair: `{workspace_id="_other",zombie_id="_other"}`.

### Cardinality

4096 composite slots. Both labels are server-assigned (workspace_id and zombie_id are UUIDs or opaque server IDs — no user-controlled strings ever become label values). Label value sets are bounded by product reality: workspaces × zombies per workspace. An operator can watch `zombie_workspace_metrics_overflow_total` — if it's non-zero, cardinality has saturated the slot table and some tokens are being aggregated under the `_other` bucket pair. That's degraded observability but not a correctness bug or an unbounded growth risk. Kishore, if you want a hard cap per tenant, we'd enforce it at admission (skill validation), not here.
