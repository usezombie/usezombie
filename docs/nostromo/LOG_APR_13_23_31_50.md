# Ripley's Log — M17_002 Worker/Executor Zombie Alignment

**Date:** Apr 13, 2026: 11:31 PM
**Branch:** `feat/m17-002-worker-executor-alignment`

## What shipped

The executor's `CorrelationContext` now speaks zombie-native. The v1 pipeline vocabulary (`run_id`, `stage_id`, `role_id`, `skill_id`) has been evicted from the executor layer — both from the in-memory struct and from the JSON-RPC wire protocol between the worker and the `zombied-executor` sidecar. `StagePayload` likewise lost its three stage-identity fields; identity now lives exclusively in `CorrelationContext` attached at `CreateExecution` time, and `StartStage` carries pure execution payload (agent config + message + tools + context).

## Decisions made

### 1. Drop `role_id` and `skill_id` entirely — don't rename

The spec offered a choice: remove or repurpose. I chose remove. Rationale: the sole real caller (`src/zombie/event_loop.zig`) was passing hardcoded literals — `.role_id = "agent"` and `.skill_id = session.config.name`. These were ceremony, not data. `session.config.name` is already reachable from `zombie_id` via the DB; carrying it in every RPC was v1 muscle memory. Nuking both fields removed ~40 lines of dead plumbing across 12 test files.

### 2. Pull `StagePayload` into the same commit — user override on scope

Initial plan kept `StagePayload` out of scope to stay surgical. User (Kishore) explicitly expanded scope mid-execution: "StagePayload i want it pulled." Same treatment — `stage_id`/`role_id`/`skill_id` deleted wholesale. The handler's only use of these fields was a single `log.info` line (cosmetic); rewrote the log line to emit `zombie_id`/`session_id` from the session's `correlation` struct instead.

### 3. `startStageBasic` deleted, not kept as wrapper

After the signature collapse, the "backward-compatible" helper became a trivial pass-through with no callers anywhere in `src/`. Surfaced it via RULE ORP sweep (which I initially forgot to run — Kishore caught it). Removed in the same commit. Classic case of leaving dead scaffolding behind after a refactor; the orphan sweep is there for exactly this reason.

### 4. Worker startup already zombie-only — spec §3.2 was a no-op

Spec claimed `worker.zig` still had "dual-mode vestiges." Read it carefully — it already spawns `worker_zombie.zombieWorkerLoop` per active zombie with no pipeline branch. The spec was written against an earlier state; someone already cleaned this up (pre-M17_001 or during M17_001). Marked §3.2 DONE with a note in the acceptance criteria explaining the pre-corrected state. Trade-off considered: pad the spec with a "verified clean" note vs. silently tick the box. Went with the note — future-me/other-agent reading the done spec deserves to know §3.2 wasn't actually executed against code.

## Dead ends / course corrections

- **First sed attempt failed on shell cwd reset.** The zsh harness resets `cwd` between `Bash` invocations, so multi-line scripts with `cd X; for f in ...; do sed ...` fragment. Switched to single-line `cd X && for ... done` pipelines. Cost ~3 minutes of confusion.
- **BSD sed word boundaries.** `\b` doesn't work on macOS sed. Local-variable renames (`const run_id = ...`) needed per-file Edit calls for the few straggler cases that generic regex didn't catch cleanly.
- **Website lint tooling missing.** `make lint` initially died with `eslint: command not found` because `ui/packages/website/node_modules/` wasn't installed in the fresh worktree. Ran `bun install`; lint then passed. Worth noting: fresh worktrees need `bun install` per JS subpackage before `make lint` is a hard gate.

## Out-of-scope findings (flagged, not executed)

Surfaced during RULE ORP sweep: `src/state/topology.zig` and `src/state/entitlements.zig` still carry their own `role_id` / `skill_id` fields for v1 `ProfileDoc` parsing. They're live code (consumed by `types/defaults.zig` and `workspace_billing/db.zig`), so deleting them unilaterally would break billing/entitlements tests. If the v1 pipeline profile-doc subsystem is also slated for teardown (likely yes, based on the broader M17 zombie migration arc), that's a separate workstream — suggested **M17_003 — TOPOLOGY_ENTITLEMENTS_V1_TEARDOWN**. Noted in the spec's "Out-of-Scope Findings" section so the next agent doesn't re-discover it.

## Trade-offs deliberately deferred

- **Grafana dashboard labels.** `executor_metrics.zig` wasn't touched because it doesn't emit per-field correlation labels today — but any dashboard keyed on `run_id` from executor log lines will now see `zombie_id` instead. Pre-v2.0, dashboards are expected to churn with data-model renames; didn't go hunt them down.
- **Protocol version bump.** Clean wire rename, no negotiation. Worker and executor ship in the same binary per the current deployment model, so no compat shim. If that assumption ever changes (separate release cadences), a `protocol_version` handshake will need adding — not worth the ceremony today.
