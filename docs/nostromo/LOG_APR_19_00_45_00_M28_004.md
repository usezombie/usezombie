## Session narrative

### Premise challenge

The inherited spec (written at M28_001 close) proposed splitting `src/zombie/config.zig` along a **type-family axis**: one new file per type family (`config_trigger.zig`, `config_skill.zig`, `config_budget.zig`). I rejected this before writing any code.

Reason: the type-family axis doesn't match the code. `ZombieBudget` is a 3-line struct. The `skill` field is a single `?[]const u8`. Those "modules" would be nearly empty — and more damningly, they'd leave the *actual* bulk of the file (parsing logic) undisturbed. The file was 448 lines; `parseZombieConfig` alone was 104 lines — already a RULE FLL-50 violation that the original spec didn't even surface.

I proposed instead a **lifecycle axis**: types / parser / markdown / validate. This matches where the lines actually live. It also forces the decomposition of `parseZombieConfig`, which was the real refactor debt.

User approved the lifecycle axis with default configuration: façade in `config.zig` for low caller churn, per-module `_test.zig` files, leave `config_helpers.zig` (319L, in cap) untouched.

### What actually happened

1. **Sibling spec** — While auditing file sizes I noticed `src/config/runtime.zig` (448L) has the identical FLL violation in an unrelated subtree. Filed M28_005 as a sibling pending spec on `main`. Independent blast radius, same pattern, same lifecycle axis.

2. **Dependency cycle I almost created** — First attempt at `config_types.zig` had the deinit call `config_helpers.freeStringSlice` and `config_helpers.freeZombieTrigger`, which would have made the import graph `config_types → config_helpers → config_types` via the existing `helpers.ZombieTrigger` aliases. Caught before writing: moved `freeStringSlice` + `freeZombieTrigger` into `config_types.zig` (type-adjacent — destructors belong with the types). Then updated `config_helpers.zig` to import `config_types` directly rather than the `config.zig` façade. Cleaner edge: façade now sits above helpers in the dep graph, not adjacent.

3. **Size-assertion miss** — Per ZIG_RULES progressive cleanup, I added `comptime { std.debug.assert(@sizeOf(ZombieConfig) == 216); }`. Guessed 216 from "9 slice fields + trigger union + budget + gates optional". Actual is 288 bytes. Flipped to `@compileLog` to get the truth, then locked in 288. Progressive cleanup earns its keep only when the asserted number comes from the compiler, not from back-of-envelope arithmetic.

4. **parseZombieConfig decomposition** — Broke the 104-line body into 8 per-field helpers (`parseNameField`, `parseTriggerField`, `parseSkillsField`, `parseCredentialsField`, `parseNetworkField`, `parseBudgetField`, `parseGatesField`, `parseExtendedFields` → which further splits into `parseSkillRef` + `parseChainArray`). Each ≤ ~20 lines. Orchestrator became 47 lines threading an `errdefer` chain per ZIG_RULES "Struct Init Partial Leak" — partial builds free every already-duped field. Added a leak-detection test (`partial-build leak check`) that proves it: submits JSON with valid name/trigger/skills/credentials but a negative `daily_dollars`, expects `InvalidBudget`, and relies on `std.testing.allocator` to panic on any missed free.

5. **Test discovery — course correction from the user.** First draft wired 8 `_ = @import(...)` lines into `main.zig` test block (one per new file + test file). User pointed out that the whole point of the façade is to collapse this. Two-part fix: (a) `main.zig` keeps a single `_ = @import("zombie/config.zig");`. (b) `config.zig` gains a `test {}` block that fans out to the 4 new `*_test.zig` files + `config_helpers.zig`. User then asked whether test imports bloat release binaries — they don't; Zig strips `test {}` blocks in non-test builds, so the pattern costs zero bytes in production. Added a short addendum to `docs/ZIG_RULES.md` codifying the façade-fan-out pattern (symlinked → lands in dotfiles).

6. **Pub audit progressive cleanup** — After removing the façade's direct calls to `helpers.freeStringSlice`, the `pub const freeStringSlice = config_types.freeStringSlice;` re-export I'd added to `config_helpers.zig` was unused. Removed per the "Progressive Cleanup" rule. Also surfaced that `zombie/webhook_verify.zig` + `zombie/config_helpers.zig` entries in `main.zig` were redundant (both already reachable via `config.zig` → `config_helpers` → `webhook_verify`). Only cleaned up the config_helpers one (webhook_verify belongs to a different module's follow-up).

### What I learned (candidate greptile entries)

- **Façade-fan-out test discovery pattern.** Recording in `ZIG_RULES.md` as the preferred alternative to per-file `main.zig` entries. Strictly better for files with a clear façade.
- **Comptime `@sizeOf` assertions only work if the number comes from `@compileLog` first.** Guessing wastes a build.
- **Type-family vs lifecycle axis for module splits.** When a type is ≤3 lines, it doesn't deserve its own module; split by lifecycle instead.
