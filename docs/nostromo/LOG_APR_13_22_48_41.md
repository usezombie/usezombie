## The flat-tyre framing (the hidden high-pain problem)

A flat tyre is usually acute, visible, blocking. Memoryless zombies don't feel flat-tyred because the pain shows up as three slow leaks, not a blowout:

1. **Death by re-steering.** Operator tells the zombie the same thing repeatedly ("don't pitch enterprise to sub-50"). Each correction feels minor. Cumulatively it's a huge tax, but no single moment looks like failure.
2. **Confident wrongness with delayed blowback.** "Following up on our last call…" when no call happened. Damage shows up days later as churn, cold leads, ghosted candidates. The cause looks like "AI wasn't good enough" rather than "AI had no memory."
3. **Silent redundancy.** Re-researching Acme. Re-classifying the same flap. Re-asking Sarah the same screen questions. Small per-instance, large in aggregate, invisible to an operator not watching closely.

**Reframed:** the zombie feels *almost* useful — enough to keep, not enough to trust. The operator becomes its permanent supervisor. Memory is what converts "needs re-briefing every run" into "remembers the briefing." That's the flat tyre. It hides because it's the difference between "no AI" and "underwhelming AI," not "no AI" and "no car."

**M14_004's second-order flat tyre:** once memory ships, you can't tell if it's working. Recall could be silently missing. Steering could be stored but not applied. Wrong-context assertions are invisible without a human feedback signal. M14_004 exists to make memory's effectiveness measurable — otherwise we've shipped a black box and hoped.

## Mechanism generalization (beyond the handful of examples)

Store/recall is deliberately boring — a keyed KV store per zombie, `(workspace_id, zombie_id, key) → JSON`. The namespaces (`candidate:`, `alert_fingerprint:`, `steering:`, `incident_sig:`) are conventions enforced by the **skill template**, not DB constructs.

- **Who picks keys:** the LLM, constrained by template-prescribed namespaces. Template says "write `candidate:<email>` at run end"; LLM fills in the identifier. This keeps discipline without being rigid.
- **Who decides recall:** the template prescribes recall points ("start of every run, recall these keys based on trigger"); the LLM composes the actual lookups from the incoming context (extract email → try `candidate:<email>`; extract thread ts → try `thread:<ts>`; list all `steering:*`).
- **Missing entries don't fail** — they return null, zombie proceeds without that context. Each hit/miss emits an event; M14_004 watches the stream.
- **No RAG, no vector search** (per existing memory feedback in `feedback_no_rag_no_search_infra.md`). Exact-key lookup with tag/category filtering; LLM reasons over the candidate set in-context. Legibility is a feature — an operator can read any zombie's memory as a flat list and understand exactly why it acted as it did.

**What makes new archetypes free:** add a new skill template with its own namespace conventions (e.g. support zombie: `customer:<email>`, `article:<slug>`, `escalation:<ticket>`). No infra changes. M14_004 metrics work out of the box because they observe the mechanism, not the content.

## Archetype-specific use cases (captured in detail)

### Lead Collector (inbound `usezombie@agentmail.to`)
- **Trigger:** email from contact form. Event-driven, one email = one run.
- **Stores:** `lead_{company_slug}` (person + company + stage + next action), `followup_{company}_{date}` (daily), `steering:*` (operator chat corrections).
- **Recalls:** `lead:<company>` on every inbound; `steering:*` always; `thread:<id>` to prevent double-sends.
- **Flat tyre hidden here:** re-researching the same company three emails in a row; pitching enterprise to sub-50 after you told it not to; saying "per our last call" to a first-time contact. Each is small; the pattern destroys trust in the tool.

### Ops Zombie (Grafana webhook / Loki poll)
- **Trigger:** alert fires or cron poll window.
- **Stores:** `incident_sig_{service}_{symptom}_{root_cause_hash}` (permanent post-mortem library), `noise_pattern_{description}`, `slo_{service}`, `steering:{service}:after_10pm`.
- **Recalls:** signature match on every alert before classifying; noise patterns for suppression; steering for time/service context.
- **Flat tyre hidden here:** paging humans at 3am for a flap the zombie has seen 47 times and already knows resolves itself. Low recall-hit rate here has human-waking consequences — this is where the metric isn't a vanity number.

### Hiring Agent (`#hiring` Slack messages)
- **Trigger:** HR message in channel. Thread continuity is central.
- **Stores:** `candidate_{name}_{role}` (profile + loop history), `role_{slug}` (rubric), `interviewer_{name}_preferences`, `steering:rejections` (legal-preferred phrasing).
- **Recalls:** candidate by email; role rubric; thread state to avoid double-drafting.
- **Flat tyre hidden here:** "Hi Sarah, great to continue from your onsite" when she never onsited. In hiring, wrong-context isn't just embarrassing — it's legal/brand risk. This is why wrong-context-asserted is the P0 trust metric across all archetypes.

## Elaborated use cases — step by step

These are the detailed walkthroughs the operator asked for, preserved verbatim so future agents don't have to re-derive them. Each shows a full run: trigger → recall → act → store → human loop → metrics.

### Use case 1 — Lead Collector (inbound)

**Real flow per operator:** contact form on `usezombie.com/contact/sales` → email to `usezombie@agentmail.to` → zombie traps the email, reads the contact, researches, drafts a reply, notifies the operator. Operator can replay in the UI and chat to steer.

**Step by step:**

1. **Trigger.** Prospect `jane@acme.com` fills the form. AgentMail webhook delivers the email to the Lead Collector zombie's webhook URL. One email = one run. No cron, no polling.
2. **Recall — before research.** Zombie extracts identifiers from the email payload and does exact-key lookups:
   - `memory_recall("lead_acme_corp")` → hit. Returns: CTO Jane, team ~200, asked about pricing three weeks ago, you marked "sub-50 push declined."
   - `memory_recall("thread:<message_id>")` → miss. New thread.
   - `memory_list(category="core", tags=["steering"])` → returns `steering:sub_50_employees` ("skip demo CTA, offer self-serve link") and `steering:tone` ("be direct, no marketing fluff").
   - Each recall emits `memory.recall.hit` or `memory.recall.miss` with `{zombie_id, key, latency_ms, archetype: "lead_collector"}` on the event stream.
3. **Act — research only what's missing.** Because `lead_acme_corp` hit, zombie skips LinkedIn scraping and company-size research. It only researches what's new: Jane's current role, recent news. Writes the research actions to `core.activity_events` (one row per tool call).
4. **Act — draft.** Zombie drafts a reply conditioned on the recalled context ("picking up where we left off on enterprise pricing, acknowledging your team size") and the steering rules ("direct tone, skip demo CTA because Acme is >50"). Draft goes to `core.activity_events` as a `draft_created` event.
5. **Notify operator.** Slack DM or dashboard notification: "Lead Collector drafted a reply to jane@acme.com. Replay to review."
6. **Store — what was learned.** On run completion, zombie writes:
   - `lead_acme_corp` updated: new interaction date, new stage ("re-engaged"), Jane's updated role.
   - `thread:<message_id>` created: `{last_action: "drafted_pricing_reply", approved_by_human: false}`.
   - Possibly a daily entry `followup_acme_corp_2026-04-20` if the draft suggests a future nudge.
   - Each store emits `memory.store.ok` or `memory.store.full` (if at cap).
7. **Operator loop — in-app replay + chat steering.** Operator opens the zombie's replay view. Sees the activity timeline (`core.activity_events`) and the memory view (`memory.memory_entries`). If the draft is wrong or tone is off, operator types: *"don't mention competitor X, she's allergic."* That chat message triggers another `memory_store("steering:acme_corp:no_competitor_mentions", ...)` — the steering persists for future Acme runs.
8. **Operator loop — thumbs-down (the wrong-context signal).** If the zombie asserted something false ("per our last call" when no call happened), operator clicks thumbs-down on that activity event. Writes a row into `memory_feedback` keyed by `(zombie_id, activity_event_id, reason)`. That row is the ONLY input to the wrong-context-asserted rate — there is no heuristic, it's pure operator signal.
9. **Aggregator (separate from the run).** Every 5 min, `memory_metrics.zig` reads the last 7 days of events + `memory_feedback` rows + `core.activity_events` (for duplicate-action detection) and rolls them into four rates keyed `(zombie_id, archetype)`. Dashboard reads the pre-computed rates; it never live-computes.

**What good looks like after a month:**
- Recall-hit rate rises from ~0% to 40-60% (most inbound leads are re-engagements or same-company follow-ups).
- Duplicate-action rate stays under 2% (zombie isn't sending the same intro twice).
- Wrong-context-asserted near zero.
- Memory size grows linearly with distinct companies contacted, then plateaus as pruning kicks in.

### Use case 2 — Ops Zombie (alert-driven)

**Real flow:** Grafana alert webhook or 5-min Loki log poll fires. Zombie classifies, posts to Slack/Discord. Critical actions (paging on-call, auto-scale) go through approval gate.

**Step by step:**

1. **Trigger.** Grafana fires `api-server error rate > 5%` webhook. Payload contains alert name, labels, current value. One alert = one run.
2. **Recall — before classification.** Zombie derives a fingerprint from the alert (service + symptom) and recalls:
   - `memory_recall("incident_sig_api_server_pg_timeout")` → hit. Returns: signature seen 47 times over last 3 months, last 3 occurrences were flaps that auto-resolved in <4 min, root cause = pg connection pool exhaustion, runbook = `scale-pool-to-50`.
   - `memory_recall("noise_pattern_api_server_batch_3am")` → hit (current time is 03:12). Returns: "batch job causes 30s error spike at 03:00-03:05 nightly — suppress unless sustained >10 min."
   - `memory_list(category="core", tags=["slo", "api-server"])` → `slo_api_server` ("99.9% target, current month 99.94% — room to absorb minor flaps").
   - `memory_recall("oncall_roster_weekly")` → `{primary: "jane", secondary: "sam"}`.
   - Each emits hit/miss events.
3. **Act — classify using the recalled context.** Armed with the fingerprint hit + noise pattern + time-of-day steering, zombie classifies this alert as `noise` (not a real incident). Without memory, it would have classified as `warning` and posted to Slack — a page for something the operator has already told it to suppress.
4. **Act — suppress or post.** Because classification is `noise`, zombie does NOT post to Slack. Instead it updates the existing aggregate count in the `incident_sig_api_server_pg_timeout` entry (occurrences + 1, last_seen = now). Writes a `suppressed_noise` event to `core.activity_events` so the operator can still see it happened.
5. **Alternate branch — if classification had been `critical`.** Zombie drafts Slack message, hits approval gate if `@on-call` tag is in the body. Operator sees `[Approve] [Deny]` in Slack DM. On approve, message posts.
6. **Store — signature update.** On run completion:
   - `incident_sig_api_server_pg_timeout` updated with new last_seen + occurrence count.
   - If this had been a NEW signature (recall miss), zombie would create a new `incident_sig_{service}_{symptom}_{hash}` entry with `{first_seen, root_cause: "unknown, investigating", runbook_applied: null}` — to be filled in post-mortem.
   - Each store emits an event.
7. **Operator loop — post-mortem steering.** After a real incident resolves, operator opens the signature entry in the memory view and edits it: adds the root cause, runbook link, and a note "this is NOT the same as the April 2 OOM, separate signature." That edit lands as a `memory_store` call and persists. Next alert with matching fingerprint gets the corrected context.
8. **Operator loop — thumbs-down.** Zombie drafted "this looks like the April 2 OOM incident" but actually it's the pg pool flap. Operator thumbs-down the draft → `memory_feedback` row → wrong-context-asserted rate ticks up. High rate here means signatures are overlapping too much; fix is usually more specific key construction.
9. **Aggregator.** Same 5-min cron. Per-archetype rollup panel on Grafana surfaces `ops_zombie` recall-hit trending up over 30 days — proves the zombie is learning your infra's personality.

**What good looks like:**
- Noise-suppression saves the on-call human from ~20 pages/week.
- Recall-hit rate > 70% at steady state (most alerts are variations of things seen before).
- Duplicate-action = 0 for destructive actions (auto-scale, restart) — approval gate + memory of "already scaled 2 min ago" prevents loops.
- Wrong-context-asserted near zero — catastrophic if it drifts, because SRE trust collapses.

### Use case 3 — Hiring Agent (Slack-driven)

**Real flow:** HR posts in `#hiring`. Zombie drafts response (interview questions, scheduling, feedback summary) and posts to the thread. Offers and rejections require per-action approval.

**Step by step:**

1. **Trigger.** HR posts: *"Sarah from the April 2 phone screen wants to schedule the technical round."* Slack Events API → webhook. One message = one run.
2. **Recall — before drafting.** Zombie extracts candidate name / email / thread context:
   - `memory_recall("candidate_sarah_smith_sr_backend")` → hit. Returns: Apr 2 phone screen by Raj (pass), strengths = Rust + distributed systems, concerns = no on-call experience, stage = `phone_screen_done`.
   - `memory_recall("thread:<slack_ts>")` → hit. Returns: previous message in this thread was HR asking "how did Sarah do?" — zombie responded 2 days ago with the phone screen summary.
   - `memory_recall("role_senior_backend_2026_q2")` → hit. Returns: rubric (systems design 40%, coding 30%, culture 30%), must-haves, hiring manager = Kishore.
   - `memory_recall("interviewer_priya_preferences")` → hit. Returns: Priya does systems design deep-dives, needs 90-min slots, available Tue/Thu afternoons IST.
   - `memory_list(category="core", tags=["steering", "rejections"])` → `steering:rejections` ("use 'strong pipeline right now,' not 'not a fit'").
3. **Act — draft with full context.** Zombie drafts: *"Scheduling Sarah for the technical round with Priya. Based on her phone screen strengths (Rust, distributed systems), Priya will focus on systems design — 90 min, Tue Apr 21 or Thu Apr 23 afternoon IST. Want me to send the calendar invite?"* Draft references the phone screen results HR would have had to re-ask for without memory.
4. **Post to thread.** Via Slack API. Writes `message_posted` event to `core.activity_events`.
5. **Approval gate branches.**
   - Scheduling messages: no gate, posts directly.
   - Offer letters / rejections: firewall rule fires → DM to hiring manager with `[Approve] [Deny]`. Memory recall of `steering:rejections` applied to the draft BEFORE the approval DM, so the manager is approving already-corrected phrasing.
6. **Store — updated candidate file.** On run completion:
   - `candidate_sarah_smith_sr_backend` updated: stage → `technical_round_scheduled`, next_action → "await Priya's feedback Apr 23."
   - `thread:<slack_ts>` updated: last_action = "proposed_technical_round_scheduling."
   - Daily entry `followup_sarah_smith_2026-04-23` created (72h after scheduled round to nudge Priya for feedback).
7. **Operator loop — in-app replay / chat steering.** Hiring manager reviews the draft in the replay UI. If they want to change the rubric emphasis ("less systems design for this candidate, she's more of an implementer"), they type the correction → stored as `steering:candidate_sarah_smith:rubric_override`. Applies on next run.
8. **Operator loop — thumbs-down (the P0 signal here).** Zombie drafted "continuing from your onsite feedback" but Sarah never onsited. Manager hits thumbs-down. `memory_feedback` row written. This is the P0 metric in hiring — because a wrong-context assertion in a rejection letter or offer is a **legal/brand risk**, not just an embarrassing draft. Wrong-context-asserted must stay near zero.
9. **Aggregator.** Per-archetype rollup shows `hiring_zombie` wrong-context trend over 30 days. If it ticks above 0.5%, operator investigates — usually means candidate keys are colliding (two Sarahs) or memory is stale (candidate dropped out but entry not updated).

**What good looks like:**
- Re-question rate drops from ~30% (no memory) to <5%. HR stops typing the same candidate context twice.
- Stage recall-hit > 90% — every HR message gets continuity instead of cold intro.
- Wrong-context-asserted ~0. Non-negotiable in hiring.
- Daily-followup compliance > 80% — Priya gets nudged for feedback in-window, not forgotten for a week.

### The generic sequence (what every archetype shares)

```
[external trigger fires — email / webhook / Slack event / cron]
    → run starts
        → memory_recall × N (one per template-prescribed lookup)
            → emit memory.recall.hit | memory.recall.miss per lookup
        → agent reasons using recalled context + incoming trigger
        → agent takes action (tool calls logged to core.activity_events)
            → approval gate branch if action is high-impact (Slack/Discord/dashboard DM)
        → memory_store × M (distilled facts learned this run)
            → emit memory.store.ok | memory.store.full per write
    → run ends

[separately, async — operator at their leisure]
    → opens replay UI
        → reads core.activity_events (timeline)
        → reads memory.memory_entries (what zombie knows)
    → optional: chat to steer → memory_store("steering:...", ...)
    → optional: thumbs-down → INSERT INTO memory_feedback

[separately, async — every 5 min]
    → memory_metrics aggregator reads last 7 days of events + memory_feedback + activity_events
    → writes rolled-up rates keyed (zombie_id, archetype)
    → dashboard panel + Grafana read the pre-computed rates
```

This separation is why adding a new archetype is free from an observability standpoint: the template prescribes the keys and the recall/store points, the rest of the pipeline is archetype-agnostic.
