## Why this workstream existed

There were two credential surfaces shipped to operators. `zombiectl credential add` wrote to `vault.secrets`, which every zombie actually read at runtime via bare-key references in `TRIGGER.md`. `zombiectl skill-secret put` wrote to `vault.workspace_skill_secrets`, which nothing on the zombie path ever read. An operator who followed their first instinct had a 50/50 chance of storing their credential in a table the zombie could not see. The docs even carried a disclaimer saying "`skill-secret` is distinct from and unrelated to the standard zombie credential flow" — the prose form of the bug.

Pre-v2.0 is the cheap window to fix it. Delete the parallel surface; keep the one zombies read.

## Decisions worth remembering

**Which helpers counted as dead with the handler.** `SkillSecretScope`, `EnvPair`, `SecretInjectionPlan`, and `buildSecretInjectionPlan` were not obviously tied to the removed table — they looked like reusable secret-plumbing utilities. Grep across the whole tree said otherwise: zero callers outside the two modules that defined them and tested them. The decision was to remove them rather than leave "reusable" code nobody reused. The audit history is in git if they come back.

**`rbac_http_integration_test.zig` narrow clean.** The test name was "RBAC endpoints enforce operator and admin roles over live HTTP" — covering skill-secret + billing + zombie-lifecycle. I kept the test (and the test name), removed only the skill-secret blocks, and preserved the unauthenticated-401 coverage by re-pointing it at the billing endpoint. The zombie-lifecycle coverage was already independent and untouched.

**Commit cadence vs. tier-3 green.** Each section committed separately — §1 could not pass `make test-integration` alone because `crypto_store.zig` still referenced the dropped table. I explicitly skipped tier 3 at §1 and §2 boundaries and ran it once at the end against a fresh DB (`make down && make up && make test-integration`). This is fine for a feature branch; PRs gate on branch-tip tier 3, not per-commit.

**Docker project collision mid-branch.** `make down && make up` failed initially because the still-running `zombie-postgres`/`zombie-redis` containers belonged to compose project `usezombie-m28-003` (another worktree). I paused and asked the user rather than stop containers owned by another in-flight workstream; they ran `make down` from their side, and I re-ran `make up` cleanly afterward. Consulted the "do not use destructive actions as a shortcut" rule — correct call.

**Bundled OpenAPI vs. source-of-truth.** First edit hit `public/openapi.json` directly and got regenerated away by `make lint`'s openapi bundle step. The real source is `public/openapi/root.yaml`. Lesson: when the bundled file has a matching source directory, always edit the source.

**VERSION drift.** On-disk VERSION was `0.21.0` but `build.zig.zon` and `zombiectl/package.json` were still at `0.9.0` — pre-existing drift from prior milestones. Bumped VERSION to `0.23.0` (matching the docs changelog label for this release) and ran `make sync-version`, which aligned all three files plus `zombiectl/src/cli.js`. `make check-version` green afterward.
