## Decisions

- **Two worktrees, two PRs.** User asked explicitly. Spec bookkeeping lives in usezombie so the milestone lifecycle stays in one repo; real doc edits live in the docs repo because that is what Mintlify deploys. The usezombie PR is the source of truth for status DONE + VERSION bump; the docs PR carries the user-visible change.
- **v0.21.0 for the docs reshape.** Pre-v1.0 breaking change → minor bump per rules. M29_002 (the code-side spec/run removal) will ride the same 0.21.0 line when it lands — both are part of the same M29 batch.
- **No redirects for `/specs/*` and `/runs/*`.** Pre-v1.0, no production traffic. Adding redirects is its own milestone if ever needed. Per the spec's Out of Scope section.
- **English verb `run` is allowed.** The spec's word-boundary regex was overreach. Amending the spec rather than rewriting every legitimate English sentence was the lower-risk move, and matches the spec's stated *intent* (remove v1 product nouns, not outlaw the English language).
- **`AGENTS.md` terminology overhaul in scope.** The docs repo's AGENTS.md mandated spec/run terminology — that file had to flip or future contributors would reintroduce the forbidden terms. Added a `Forbidden terminology` block that lists the v1 nouns.
- **Zombies pages carry their own depth.** Rather than thin stubs that defer to the CLI reference, each zombies page is self-contained — SKILL.md authoring, credential firewall diagrams, webhook authentication matrix, template anatomy. The goal was that a user who lands on any single page gets enough context to act.
