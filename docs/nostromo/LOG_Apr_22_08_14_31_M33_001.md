## Pivot: option B — natural-language policy

The user challenged the per-tool sub-skill layer immediately after the
first parked commit: "in homelab you have invented a tool layer
(kubectl ...README.md)? is that needed? why would I need that? can I
not keep it in the skills.md? or tools or a simpler natural language
way?"

**Honest reassessment.** The sub-skill split I landed was spec-driven,
not consumer-driven. The homelab zombie is the only thing that uses
`kubectl-readonly` and `docker-readonly` today. Future-reuse is
speculative. YAGNI applies.

**Offered three options:** (A) inline `tools[]` allowlist in the
flagship SKILL.md, (B) natural-language allowlist in the prompt body
only, (C) keep the sub-skills. Recommended A; user picked B.

**What changed:**

- Deleted `samples/homelab/skills/` (both sub-skill SKILL.md files
  and the directory).
- Rewrote `samples/homelab/SKILL.md` — added a "Tools you can use"
  section with per-tool allowed/forbidden lists in prose. Kept the
  lists on single lines with "Never use" so grep gates stay robust
  against substring false positives (the first attempt got bitten by
  "dis**patch**er" matching `patch`, "re**start**" matching `start`,
  and so on — fixed with `\b` word boundaries on grep + merged lines).
