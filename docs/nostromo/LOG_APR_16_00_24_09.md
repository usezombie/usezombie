## How we got to M24_001

The concrete gaps were:

- Routes documented in §6.1 of M12_001's spec — `GET /v1/workspaces/{ws}/zombies`, `POST /v1/workspaces/{ws}/zombies/{id}:stop`, `GET /v1/workspaces/{ws}/activity`, `GET /v1/workspaces/{ws}/firewall/metrics`, `GET /v1/workspaces/{ws}/firewall/events`, `GET /v1/workspaces/{ws}/api-keys` — none of them existed in `router.zig` or `router_test.zig`. The flat `/v1/zombies/*` routes had been the convention since M2_001. Workspaces had been added later (M9 and M16) as path-scoped routes, but the zombie family never migrated.
- Firewall metrics/events were not persisted anywhere — `outbound_proxy.zig` processed requests but did not write to any audit table.
- The kill switch the UI needed did not exist; only workspace-level pause/resume existed.
- Trust score was not a real data primitive — not in any schema file, not computed anywhere.
- API keys: `core.tenants.api_key_hash` is one hash per tenant. The multi-key management the Settings page assumed (named keys, revocation, rotation) required a new table that did not exist.

Per AGENTS.md's **Spec-vs-rules conflict check**, I surfaced all of this in PLAN and offered three paths: amend the M12 spec down to what was buildable, expand M12 to build the missing backend, or stub the pages that had no data. The user chose expansion — and pushed for a REST-ful API redesign ("workspace uuid in path, query params only for page/search/etc.").

At that point the work split naturally into two PRs. M24_001 is PR #1: the mechanical URL refactor, independent of UI. M12_001 stays in `docs/v2/active/` and depends on M24_001 landing first; its PR #2 will be built off the post-refactor main and drop the firewall page + trust score + multi-key settings to follow-up milestones. ## Design decisions during M24_001

### Handler pattern: path-arg, not query

The original handlers read `workspace_id` from `qs.get("workspace_id")` or from the POST body. Both were identity-in-query-param or identity-in-body, which RULE RAD §4 forbids. The new pattern, consistent with `external_agents.zig` (RULE WAUTH reference implementation):

```zig
pub fn innerFoo(hx: Hx, workspace_id: []const u8, ...) void {
    // 1. validate workspace_id shape (UUIDv7 canonical hex)
    if (!id_format.isSupportedWorkspaceId(workspace_id)) { ... fail ... }
    // 2. acquire conn
    const conn = hx.ctx.pool.acquire() catch { ... internalDbUnavailable ... };
    defer hx.ctx.pool.release(conn);
    // 3. RULE WAUTH
    if (!common.authorizeWorkspace(conn, hx.principal, workspace_id)) { ... fail 403 ... }
    // 4. do the work, scoping every DB query by workspace_id
}
```

This guarantees two properties: (a) a cross-workspace path (`/v1/workspaces/A/...` from a token scoped to workspace B) returns 403 at the middleware-like check, not after data is read or written, and (b) every DB query has `workspace_id` in its WHERE clause, so even if authorization is bypassed the query cannot touch another tenant's rows. Defence in depth.

One subtle choice: for zombie-child resources (activity, grants, `:steer`), the handler also verifies the zombie's actual `workspace_id` matches the path `workspace_id`. Mismatch returns **404, not 403**. The reasoning is the same as external_agents' RLS path: 403 leaks the existence of the zombie across workspaces; 404 says "this path resolves to nothing from your vantage point." The cost is a second DB read to look up the zombie's workspace; acceptable because it's always a primary-key lookup and it's only on mutation paths.

### Matcher naming and generalization

I inherited three ad-hoc matchers — `matchZombieId`, `matchZombieSuffix`, `matchZombieGrantRevoke` — and one generic one (`matchWorkspaceSuffix`). For the workspace-scoped refactor I added:

- `matchWorkspaceZombie` — returns `WorkspaceZombieRoute { ws_id, zombie_id }` — for DELETE
- `matchWorkspaceZombieSuffix(path, suffix)` — generic, also returns `ZombieTelemetryRoute` (same shape). I considered renaming `ZombieTelemetryRoute` to `WorkspaceZombieRoute` since they are structurally identical, but I left the historical name to avoid a mass-rename of `zombie_telemetry.zig` as part of this refactor.
- `matchWorkspaceZombieGrant` — returns `WorkspaceZombieGrantRoute { ws_id, zombie_id, grant_id }` — for grant DELETE
- `matchWorkspaceZombieAction(path, action)` — for colon-action paths like `:steer`. Added during slice 6a for the M23 `:steer` migration.

After the M23 merge, `matchZombieAction` was deleted and `matchZombieId/Suffix/GrantRevoke` were also removed (zero callers post-migration). This was caught by the orphan sweep.

### Route match order

Zig's `match(path)` is a chain of `if` statements; the first one that returns wins. Ordering matters because some of the matchers' shapes overlap. The rule I followed: **most specific first, least specific last**. For the zombie family:

1. `matchWorkspaceZombieAction(path, ":steer")` — colon-action, most specific
2. `matchWorkspaceZombieSuffix(path, "/activity")` — suffix child
3. `matchWorkspaceZombie(path)` — plain `/zombies/{id}` — would happily match `/zombies/{id}:steer` if asked before the action matcher
4. `matchWorkspaceSuffix(path, "/zombies")` — collection

I initially had `matchWorkspaceZombieAction` later in the list. The M23 test that survived the merge (`test "match resolves zombie_steer route"`) caught the collision — `matchWorkspaceZombie` returned `{ ws_id, zombie_id: "z1:steer" }` because it treats `:steer` as just part of the single segment (no `/` in it). Reordered, test passed.

### Pre-v2.0 API removal policy (RULE EP4)

`cat VERSION` = `0.9.0` at session start (and stayed there — this refactor doesn't bump the version). Under RULE EP4's pre-v2.0 carve-out, removed flat endpoints **return bare 404**, not 410 Gone. I asserted this explicitly in the router tests (`try std.testing.expect(match("/v1/zombies/") == null)`) so the policy decision is visible in the test log, not implicit. When we cross v2.0 we'll have to add 410 Gone stubs for any remaining deprecations; that's a separate milestone.

### Body cleanup (RULE RAD §4)

`CreateBody` (zombie create) and `CredentialBody` both carried `workspace_id` fields. Both were removed. The corresponding `validateCreateFields` / `validateCredential` functions also lost the workspace_id check. The API now rejects the field silently (via `ignore_unknown_fields = true` on JSON parse) rather than 400-ing on it — old clients that still send it continue to work until they update, but the URL path is authoritative and any mismatch is ignored.

At the end of slice 5, the user asked to pull origin. M23_001 (zombie steer, live steering via Redis interrupt) had merged to main during my slice work. Origin was 9 commits ahead, branch 6 commits ahead — a real merge was needed.

- **Keep M23 additions** (zombie_steer route enum variant, zombie_steer_http.zig handler, `matchZombieAction` matcher)
- **Defer M23 :steer migration to slice 6a** — kept as flat `/v1/zombies/{id}:steer` through the merge, migrated immediately after. This was a deliberate scope-limiting decision: mixing rebase resolution with new migration is confusing; do them in separate commits.

One broken test after merge: M23's `test "match resolves zombie_steer route"` had a sub-assertion that `match("/v1/zombies/{id}")` (flat plain-id) returned a variant that was NOT `zombie_steer`. Under M24 that path now returns null. The `.?` unwrap panicked. Fixed by replacing the sub-assertion with `try std.testing.expect(match(...) == null)` — asserting the removal is what the new semantics demand.

Slice 6a then properly migrated `:steer` to workspace-scoped. Matcher renamed, enum variant renamed, handler gets `workspace_id` path arg, `authorizeWorkspace` added, and `resolveZombieExecution` compares zombie's workspace to the path workspace (404 on mismatch). Integration tests updated: T1.4 (cross-workspace zombie) now puts the caller's ws_id in the URL with a zombie from another workspace — server returns 404.

## zombiectl CLI migration

zombiectl was factored around a single `ZOMBIES_PATH = "/v1/zombies/"` constant with call sites building query strings. The refactor replaced that with named helpers in `api-paths.js` — one function per route shape:

```js
export const wsZombiesPath = (wsId) => `${WORKSPACES_PATH}${enc(wsId)}/zombies`;
export const wsZombiePath = (wsId, zombieId) => `${WORKSPACES_PATH}${enc(wsId)}/zombies/${enc(zombieId)}`;
// ... wsZombieActivityPath, wsCredentialsPath, wsGrantsListPath, wsGrantPath
```

Six call sites updated in `zombie.js`, two in `grant.js`. One CLI contract change: `zombiectl logs` previously did not require a zombie id (the server interpreted it as workspace-wide activity because the handler only scoped by query `zombie_id`). Post-refactor the activity endpoint is per-zombie, so `zombiectl logs` now requires `--zombie <id>`. Added a new negative test for the missing-flag path to make the regression explicit.

One thing I intentionally did NOT fix: `zombiectl kill` (no name) tries to DELETE the collection, which never had a server handler — 405 Method Not Allowed. Pre-existing bug. Left as-is because the URL shape is updated and the bug is out of M24's scope.
