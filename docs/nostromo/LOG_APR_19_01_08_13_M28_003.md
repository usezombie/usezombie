## What I set out to do

Turn `public/openapi.json` from a 4,323-line hand-edited JSON blob into a build artifact produced by bundling split YAML sources, and add CI gates that catch drift between the Zig router, the OpenAPI spec, and the Mintlify documentation. The user's bigger concern was that autonomous agents needed to reliably edit the spec (rename paths, append paths, update descriptions) without corrupting 4K+ lines of nested JSON.

## What actually happened

### The spec was wrong about three things. I amended before executing.

The first-draft spec prescribed a regex-scrape of `router.zig`'s `match()` body to produce the route list for the sync gate. I opened `src/http/router.zig` expecting to confirm this worked and instead found routes dispatching through `std.mem.eql` + `std.mem.startsWith` + helpers like `matchWorkspaceSuffix(path, "/credentials/llm")` ‚Äî there are essentially no `"METHOD /path" => Route` string arms to scrape. Any regex we wrote would miss more than half the routes.

I amended ¬ß3 to introduce `pub const route_manifest` in a new `src/http/route_manifest.zig` ‚Äî an explicit list of 57 `(method, path)` pairs. Two reasons this beats the regex approach: (a) it gives a single canonical place to read the full public URL surface, (b) the Python sync script becomes a five-line regex over a stable data structure rather than a fragile scrape of hand-written dispatch code.

The second mismatch: the spec said Redocly's built-in rules would "subsume" `scripts/check_openapi_errors.py`. They don't ‚Äî Redocly's `operation-4xx-response` just checks that a 4xx exists, it says nothing about media type, ErrorBody fields, or the `/readyz 503` allowlist this codebase relies on. Porting those to a custom Redocly plugin would be strictly more code than keeping the 130-line Python script. So I kept it, and made `make openapi` call it after Redocly lint. The standalone `make check-openapi-errors` target was removed and `lint-zig` no longer depends on it ‚Äî the script now lives on the OpenAPI lane, where it belongs semantically.

The third mismatch: the spec implicitly assumed the current spec would pass Redocly lint out of the box. It doesn't ‚Äî the codebase has 11 `nullable: true` usages (OpenAPI 3.0 syntax in a 3.1.0 spec) and two pre-existing tag bugs (`"Zombie"` singular on `/stop`; undeclared `"Activity"` on `/v1/workspaces/{id}/activity`). I fixed the tag bugs in a separate narrow commit (¬ß1.4) so the ¬ß1 byte-parity claim remained auditable; the `nullable` issue is out-of-scope content-compliance work, pinned off in `.redocly.yaml` with a comment naming the follow-up.

Surfaced a fourth mismatch late during ¬ß3: `/v1/webhooks/svix/{zombie_id}` was dispatched by the router (M28_001 ¬ß5) but never documented in OpenAPI. The sync gate I was about to wire would have failed the PR. I added the operation to `public/openapi/paths/webhooks.yaml` with the Svix header parameters (`svix-id`, `svix-timestamp`, `svix-signature`) and a 202 response ‚Äî a content-addition, but a narrow one.

### The mechanical split worked first try on byte-parity after one layout fix

Initial attempt at splitting components had each external file as a flat map (`{Foo: ..., Bar: ...}`). Redocly's bundle failed because internal `$ref: '#/components/schemas/X'` inside those files couldn't resolve against their own document root. Fixed by wrapping each components file under `{components: {<section>: {...}}}` so internal refs stay self-consistent. Paths files stay flat at top level; their cross-file refs to components are rewritten by the splitter to `../components/<section>.yaml#/...` form, and Redocly rewrites them back to bundled-doc form on bundle.

After that fix, `jq -S` canonical diff between pre-split and mechanical-split bundle was empty on the first real run. Dim 1.3 satisfied.

### The in-file manifest test

The spec said to add a unit test asserting every `route_manifest` entry is dispatchable through `match()`. I implemented this by substituting every `{param}` with a single-character placeholder `"x"` and calling `match(concrete_path)`. All 57 entries dispatch; the test catches drift where someone adds a manifest entry without a corresponding `match()` arm. This sits in `src/http/router_test.zig` (test-exempt from FLL) rather than `router.zig` itself (345L, 5 lines under the FLL limit ‚Äî extracting the manifest to a separate file kept router.zig under the gate).

### The rehearsals (¬ß5) as drift-detection tests, not new-feature rehearsals

Full rename/append rehearsals would have required introducing a throwaway route through router.zig, server.zig, and a handler ‚Äî invasive for a verification step. Instead I rehearsed the Dim 3.2 and 3.3 negative paths:

- **Manifest-ahead-of-spec:** added a `GET /v1/_probe` to `route_manifest.zig`, ran `python3 scripts/check_openapi_sync.py` ‚Üí exit 1, message `"server has (method, path) not in public/openapi.json: GET /v1/_probe ‚Üí fix: add the operation to public/openapi/paths/<tag>.yaml and run 'make openapi'."` Reverted. Re-ran ‚Üí `OK: 57 method/path pairs`.
- **Spec-ahead-of-router:** added a fake `/v1/_rehearsal_probe` to `public/openapi.json` via `jq`, ran sync ‚Üí exit 1, message `"public/openapi.json has (method, path) not in server: GET /v1/_rehearsal_probe ‚Üí fix: add the route to src/http/route_manifest.zig AND wire match() + handler in router.zig / server.zig."` Reverted.

The rename/append workflows themselves are verified by construction: `make lint-openapi` runs bundle + Redocly + error-schema + sync on every commit, and the recipes in `public/openapi/AGENTS.md` are exactly the series of commands `make lint-openapi` needs green. If a recipe step is missing, `make lint-openapi` fails with the specific rule/file/line that caught it. The PR description calls this out so reviewers know what was and wasn't end-to-end rehearsed.

### Decisions deliberately not made

- **No `public/openapi.json` content compliance pass.** Fixing the 11 `nullable: true` usages and suppressing the `no-unused-components` warning for `Transition` belongs to a separate PR ‚Äî my `.redocly.yaml` explicitly names both as out-of-scope, with `struct: off` and `no-unused-components: off` and a commented pointer to why. Flipping them to `error` in a follow-up is a one-line change once the underlying content is fixed.
- **No `VERSION` bump.** M29 has already staked `0.21.0` on its branch with a breaking changelog entry. M28_003 is internal-only tooling. Per CLAUDE.md the internal-only carve-out allows folding into the next user-visible release. Whoever merges second decides the version. The PR description says so.
- **No `make openapi-add-path` or `make openapi-rename-path` helper.** The `public/openapi/AGENTS.md` recipes already encode the discrete steps an agent or human needs to run, and the user's earlier feedback was "don't create new make targets without a clear, distinct caller need." The meta-lane `make lint-openapi` is the only addition.
