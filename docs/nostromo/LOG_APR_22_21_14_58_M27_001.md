## Decisions worth capturing

- **Scope expansion captured explicitly.** The original spec asserted "no new API endpoints, no new primitives, no schema changes." Reality shipped two new endpoints, five design-system primitives, a Server Action, and an auth abstraction. Rather than pretend that claim still held, I added §13 ("Shipped beyond original scope") to the spec. This keeps the spec honest about what actually landed instead of letting the cover-page description diverge from reality. Future retros can reason from the spec alone.
- **`active_workspace_id` cookie, not JWT claim.** The switcher needed a way to persist the operator's current workspace without round-tripping Clerk every flip. Because `authorizeWorkspace` is tenant-scoped (any workspace inside the caller's tenant passes), a cookie read by the Server Action is sufficient — the JWT's `workspace_id` metadata stays a default hint, not an access lock. This is why the switcher is a cookie write and not a sign-out / re-issue dance.
- **Server-side `?search=` deferred intentionally.** §3.0 originally called for server-side search. The operator and I discussed and decided: client-side filtering on the paginated slice is adequate for alpha-volume workspaces (every alpha operator has << 100 zombies), and the backend change would have pulled in FTS index decisions that don't belong in a UI milestone. Filed as a pending follow-up spec.
- **Per-zombie billing routes stay absent.** The spec had §4 dimensions for a SpendPanel; those were quietly dropped because the `/v1/workspaces/{ws}/zombies/{id}/billing/summary` route was removed pre-v2.0 per M10-era policy and the router actually has a regression test ("rejects removed workspace billing routes — pre-v2.0 404s"). Dashboard billing display uses `/v1/tenants/me/billing` instead. Revisit when the billing UX milestone lands.
- **Dashboard integration test: unique IDs, not DELETE cleanup.** The previous main.zig had a four-line NOTE explaining why this test file was *not* wired up — the cleanup pattern conflicted with an append-only trigger on `core.activity_events`. Switching to per-call unique IDs for zombies / activity events (workspace_id + tenant_id stay fixed to match JWT tokens) eliminates the need for any in-run cleanup; `make down && make up` covers cross-run isolation. The NOTE is gone, the test is wired, and it passes clean.
