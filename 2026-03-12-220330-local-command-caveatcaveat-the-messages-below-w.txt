‚Źļ You're right to zoom out. The worker today is wearing three hats:

  1. Scheduler ‚ÄĒ claims work (SELECT FOR UPDATE SKIP LOCKED from Redis/Postgres)
  2. Executor ‚ÄĒ runs the Echo‚ÜíScout‚ÜíWarden pipeline
  3. Fault recovery ‚ÄĒ reconciles stale outbox, retries, dead-letters

  lifecycle. If no worker restarts, stale rows sit forever. If you scale down
  workers, nobody reconciles.

  The cleaner split:

  zombied serve     ‚Üí API (unchanged)
  zombied worker    ‚Üí claim + execute only
  zombied reconcile ‚Üí periodic fault recovery (new subcommand or background
  loop)

  Two approaches that fit your current architecture:

  Option A: Background reconciler goroutine inside worker
  - Worker spawns a separate thread that runs reconcileBatch on a timer (e.g.
  every 30s)
  - Still colocated with worker, but decoupled from startup/execution lifecycle
  - Runs continuously, not just at boot ‚ÄĒ catches stale rows from crashes,
  timeouts, network blips
  - Simple: no new binary, no new deploy target

  Option B: zombied reconcile as a standalone subcommand
  - Runs as a separate process (cron job, k8s CronJob, or systemd timer)
  - Full separation of concerns ‚ÄĒ executor never touches reconciliation
  - Can run on a different cadence, different machine, different resource
  profile
  - Matches the pattern of zombied migrate ‚ÄĒ operational concern, not runtime

  For v1 with single-digit workers, Option A is pragmatic ‚ÄĒ one timer thread
  inside the worker, batched with SKIP LOCKED so multiple workers share the load
   naturally. The startup-only reconcile becomes just the first tick of that
  timer.

  Option B is the v2 move when you have enough workers that you want independent
   scaling and operability.

  Which direction do you want to go?

  Now on your second question ‚ÄĒ sandbox capacity constraints. This is a real
  gap. Right now NullClaw runs directly on the worker host. The architecture doc
   mentions Landlock (filesystem) but nothing limits CPU, RAM, or disk per agent
   execution. A runaway agent can OOM the worker or fill the disk.

  Here's my take on the approach for a new milestone:

  Sandbox Resource Governance (proposed M5 or M6)

  The problem has three layers:

  1. Per-execution limits ‚ÄĒ cap what a single agent run can consume
    - cgroups v2 (Linux): memory.max, cpu.max, io.max ‚ÄĒ zero-overhead,
  kernel-native
    - tmpfs-backed workspace with size limit for disk

  3. Fleet-level scheduling ‚ÄĒ which worker gets which run

    - Future: workers advertise capacity; scheduler assigns based on fit

   v2/v3 territory.
